第5章 実践への示唆 — 何をどう変えるか

第1章〜第4章で、AIシステムの構造を理解した。ステートレス性（第1章）、確率的制御と決定論的制御の二重構造（第2章）、Harnessという設計概念（第3章）、そしてそれらを統合した基盤構造図（第4章）。本章では、この構造的理解を日常のAI利用にどう活かすかを述べる。

5.1 よくある問題と、その構造的原因

AIを使っていて遭遇する問題の多くは、構造を知れば原因が特定できる。

よくある不満構造的原因対処の方向性
同じ指示で違う結果が出る生成プロセスが確率的温度パラメータの調整、出力形式の制約
ルールを守ってくれないテキスト指示は確率的制御Hooksの導入、または許容する設計
長い会話で最初の指示を忘れるコンテキストウィンドウの上限重要指示の繰り返し、外部ファイル化
昨日の話を覚えていないステートレス設計メモリ機能の活用、状態の外部永続化
途中で手順を飛ばす確率的に指示をスキップ決定論的チェックポイントの設置

この表を眺めると、一つの共通パターンが見えてくる。ユーザーの不満の多くは、確率的なシステムに決定論的な振る舞いを期待していることに起因する。 「必ず〇〇してほしい」という期待と、「確率的にしか〇〇しない」というアーキテクチャの間のギャップだ。

ここで重要なのは、対処法が二つに分かれることだ。一つは期待を調整すること——「100%ではないが95%で十分」と割り切る。もう一つは仕組みを変えること——確率的制御ではなく決定論的制御を使う。どちらが適切かは、その作業における「失敗のコスト」で判断する。

アイデア出しで出力がばらつくのは、むしろ歓迎すべきだ。「新商品の名前を10個考えて」と頼んで、毎回同じ10個が返ってくるなら、2回目以降に聞く必要がない。ばらつくからこそ、何度も試す価値がある。一方、データ処理で手順を飛ばすのは致命的だ。入力データが欠損した状態で次のステップに進めば、以降の作業がすべて無駄になる。

5.2 問題の診断 — Harnessか、モデルか

第3章で述べた「馬と馬具」の区別は、問題が起きたときの診断にも使える。

「AIの出力がおかしい」と感じたときの判断フロー:

問題が発生
  │
  ├─ コンテキストに必要な情報が入っているか？
  │    No → Context Engineeringの問題（Harness側）
  │          対策: インプットを改善する
  │
  ├─ 情報は入っているが、指示通りに動かない？
  │    → 確率的制御の限界（Harness側）
  │      対策: Hooksの導入、または許容する設計
  │
  ├─ Harnessを整えても改善しない？
  │    → モデルの能力限界（モデル側）
  │      対策: モデルを変える、またはタスクを分割する
  │
  └─ 特定の言語・専門分野で精度が低い？
       → モデルの訓練データの偏り（モデル側）
       対策: その分野に強いモデルを選ぶ

多くの場合、問題はHarness側にある。同じモデルでも、コンテキストに載せる情報を変えるだけで出力が劇的に改善することは珍しくない。「このAIは使えない」と結論づける前に、まずHarness側でできることが残っていないか確認する。

逆に、Harnessを十分に整えた上でもモデルが特定のタスクをこなせない場合——たとえば、ある専門分野の知識が訓練データに含まれていない場合——は、プロンプトをいくら工夫しても改善しない。その場合は素直にモデルを変えるか、人間が補完する判断が必要だ。

「まずHarnessを疑い、それでもダメなら初めてモデルの限界を疑う」 という順序が、効率的な問題解決のための原則になる。

5.3 自由度の配置という設計思考

第2章で述べた「確率的制御と決定論的制御」の使い分けは、実践上は「自由度の配置」という設計思考に集約される。

LLMの出力が確率的であることは、弱点であると同時に強みでもある。曖昧な入力から意味のある出力を生成できるのは、この確率的な自由度のおかげだ。全てを決定論的に固めてしまえば、LLMを使う意味がなくなる。

重要なのは、自由度を一律に排除も放任もせず、「どこで自由にして、どこで固めるか」を意識的に設計することだ。

具体的に考えてみよう。「レポートを書いて」というタスクでは、以下のように自由度を配分できる。

自由にする部分（確率的制御で十分）:
  ・文章の表現、言い回し
  ・論点の優先順位
  ・例の選び方

固める部分（決定論的制御が望ましい）:
  ・出力フォーマット（Markdown、セクション構成）
  ・引用のルール（ソースを必ず明記）
  ・文字数の上限

「固める」手段も段階がある。フォーマットの指定はプロンプトに書けば高確率で従う（確率的制御で十分）。引用ルールは重要なので、出力後にスクリプトで検証する方が確実だ（決定論的制御の併用）。

この配分の判断こそが設計行為であり、プロンプトの文言を調整することよりもはるかに影響が大きい。

5.4 ツール選択の指針

自由度の配置を実践するには、適切なツールを選ぶ必要がある。現時点でのAIツールは、大きく三つのカテゴリに分かれる。

チャットUI（Claude.ai, ChatGPT等）: ほぼすべてが確率的制御の世界だ。ユーザーが決定論的制御を設計する手段はない。その代わり、対話の自由度が最も高い。第3章で述べた「思考をぶつける」対話には最適だ。構想を練る、設計を議論する、文章を推敲する——こうした探索的な作業に向いている。

エージェントツール（Claude Code, Codex, Gemini CLI等）: Hooksなどの決定論的制御を設計できる。ファイル操作、コード実行、外部ツール連携が可能。手順の遵守が重要な作業——実装、データ処理、定型的なパイプライン——に向いている。ただし、第9章で述べるように、ツールごとにHooksの対応状況が異なる。

API + 自作オーケストレーション: 最も自由度の高いHarness設計が可能だ。イベントループ全体を自分でコードとして書くため、確率的制御と決定論的制御の配分を完全に制御できる。ただし技術的コストが高く、ソフトウェア開発の知識が必要になる。

全部をチャットUIでやろうとするのは、ネジを回すのに金槌を使うようなものだ。逆に、アイデア出しにエージェントツールを使うのも非効率である。タスクの性質に応じてツールを選ぶ——これも「自由度の配置」の一形態だ。

5.5 構造化しすぎないこと

最後に、逆方向の注意を述べておく。

本稿はAIシステムの構造を理解し、Harnessを設計することの重要性を説いている。しかし、あらゆる作業を構造化すべきだと言っているわけではない。

チャットUIで「ちょっとこれ調べて」「この文章を直して」と頼むような日常的な使い方に、Harnessは不要だ。proposal.mdもspec.mdも要らない。必要なのは、AIに十分な文脈を与えることだけだ。

構造化が必要になるのは、以下の条件が揃ったときだ。

作業が複数セッションにまたがるとき。 ステートレスなモデルにセッション間の文脈を渡すには、外部永続化（ファイル書き出し）が必要になる。これはHarnessの最も基本的な機能だ。

手順の遵守が重要なとき。 確率的制御では不十分で、決定論的制御（Hooks）が必要な場合。データ処理パイプラインや本番環境への操作など。

成果物の品質を一定水準以上に保つ必要があるとき。 一回限りの使い捨てなら多少の品質のばらつきは許容できる。繰り返し使う成果物なら、検証の仕組みが必要になる。

これらの条件に当てはまらないなら、構造化のオーバーヘッドは作業を遅くするだけだ。「困ったら入れる」 ——構造化は、必要性を感じてから導入すれば十分である。

実用上のポイント: 「ちょっとこれ調べて」「この文章を直して」にHarnessは要らない。構造化を検討するのは、「前回のセッションの文脈が消えた」「AIが手順を飛ばした」「同じミスが繰り返される」といった具体的な問題に遭遇してからでよい。問題が起きていないのに予防的に構造化するのは過剰設計だ。
