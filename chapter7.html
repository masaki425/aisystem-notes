第7章 実践 — シンプルな課題で機能を組み立てる

第6章では、Harnessを構成する個々の機能——CLAUDE.md、Rules、Skills、Hooks、MCP、Agents——を個別に解説した。しかし、部品の説明だけでは、実際にどう組み合わせるかは見えてこない。

本章では、一つのシンプルな課題を題材に、機能をゼロから一つずつ追加していく。各ステップで「何が変わったか」「何がまだ変わらないか」を観察することで、確率的制御と決定論的制御の役割分担を体感できるようにする。

7.1 課題の設定

題材は「週次メモから構造化レポートを生成する」タスクだ。

チームの各メンバーが書いた週次メモ（箇条書きの雑多なメモ）を入力として、カテゴリ別に整理された週次レポートを生成する。入力は input/ ディレクトリにある複数のテキストファイル、出力は output/report_YYYY-MM-DD.md とする。

この課題を選んだ理由は三つある。技術的な前提知識がほぼ不要なこと、入力と出力が明確なこと、そして機能を追加するたびに改善が見えやすいことだ。

7.2 Step 1 — CLAUDE.mdだけで動かす

最小構成から始める。プロジェクトルートに CLAUDE.md を一つ置くだけだ。

# プロジェクト概要

週次メモから構造化レポートを生成するプロジェクト。

## ディレクトリ構成
- input/    — メンバーの週次メモ（テキストファイル）
- output/   — 生成されたレポート

## 作業の前提
- input/ 内のファイルは読み取り専用として扱う
- レポートは「進捗」「課題」「来週の予定」の3カテゴリに分類する

これだけで、AIに「週次メモからレポートを作って」と指示すれば、それなりのレポートが生成される。

観察すべきポイント: 同じ指示を2回実行してみよう。おそらく、カテゴリの並び順が変わったり、要約の粒度が異なったりする。「input/は読み取り専用」と書いてあるのに、AIがinput/のファイルを整理しようとすることもありうる。これが確率的制御の揺らぎだ。

この時点で得られるもの: AIがプロジェクトの全体像を把握し、概ね意図に沿ったレポートを生成する。CLAUDE.mdだけでも「だいたいうまくいく」——これが確率的制御の強みだ。

この時点で得られないもの: 毎回同じ品質のレポートが出る保証はない。手順をどの順番で進めるかはAIの判断に委ねられる。

7.3 Step 2 — Rulesを追加する

CLAUDE.mdが「地図」なら、Rulesは「交通ルール」だ。AIの行動にもう少し具体的な規律を加える。

# ファイル: .claude/rules/workflow.md

## レポートのルール
- カテゴリの順序は「進捗」「課題」「来週の予定」の固定順とする
- 各カテゴリに該当するメモがない場合は「該当なし」と明記する（空欄にしない）
- メモの内容を要約する際は、固有名詞と数値を省略しない

## 禁止事項
- input/ 内のファイルを変更・削除しない
- 出力ファイル名のフォーマット（report_YYYY-MM-DD.md）を変えない

何が変わるか: カテゴリの順序が安定する。「該当なし」が明記されるようになる。固有名詞の省略が減る。Rulesによって確率的制御の精度が上がった。

何がまだ変わらないか: 2回実行して完全に同じ出力にはならない。「input/を変更するな」と書いてあっても、確率的に守られない可能性はゼロにはならない。Rulesは確率を上げるが、保証はしない。

7.4 Step 3 — Skillsで手順を標準化する

ここまではAIに「レポートを作って」と自由に任せていた。Skillsで手順を明示化する。

# ファイル: .claude/skills/report/SKILL.md

---
name: report
description: 週次メモから構造化されたレポートを生成する。
---

## 手順
1. input/ の全テキストファイルを読み込む
2. 各メモの内容を「進捗」「課題」「来週の予定」に分類する
3. カテゴリごとに統合し、重複を除去する
4. 各カテゴリのサマリーを3行以内で作成する
5. output/report_YYYY-MM-DD.md としてレポートを出力する
6. 「レポート生成完了」と報告する

何が変わるか: 手順が明示化され、AIは「まず全ファイルを読む → 分類する → 統合する → サマリーを作る → 出力する」という順序で進むようになる。手順2を飛ばしていきなりレポートを書き始める、といった「ショートカット」が減る。再現性が向上する。

何がまだ変わらないか: 手順5の「output/に出力する」前に、手順4のサマリー作成をスキップすることは起こりうる。Skillsはテキストとしてコンテキストに入るだけなので、各手順の実行は保証されない。特に、手順6の「報告して停止」はAIがさらに作業を続けてしまう場合がある。

7.5 Step 4 — Hooksで手順遵守を保証する

ここで初めて、確率的制御の外に出る。

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "command": "bash .claude/hooks/check_report.sh"
      }
    ]
  }
}
# ファイル: .claude/hooks/check_report.sh
#!/bin/bash

errors=""

# レポートファイルが存在するか
report=$(ls output/report_*.md 2>/dev/null | head -1)
if [ -z "$report" ]; then
  errors="${errors}レポートファイルが output/ に生成されていません。\n"
fi

# 3つのカテゴリが含まれているか
if [ -n "$report" ]; then
  for section in "進捗" "課題" "来週の予定"; do
    if ! grep -q "$section" "$report"; then
      errors="${errors}「${section}」セクションがレポートに含まれていません。\n"
    fi
  done
fi

# input/ が変更されていないか
if [ -n "$(git diff --name-only -- input/)" ]; then
  errors="${errors}input/ 内のファイルが変更されています。\n"
fi

if [ -n "$errors" ]; then
  echo "以下の問題があります："
  echo -e "$errors"
  exit 1
fi

exit 0

何が変わるか: これは質的な変化だ。AIが「完了です」と宣言しても、レポートが存在しなければ差し戻される。3カテゴリが欠けていれば差し戻される。input/が変更されていれば差し戻される。確率的な揺らぎがどうであれ、最終的な成果物は必ず条件を満たす。

何がまだ変わらないか: レポートの内容の質はHooksでは検証できない。「進捗」セクションが存在することは確認できるが、その中身が適切かどうかは確認できない。意味的な判断は確率的制御（CLAUDE.md、Rules）に委ねられたままだ。

ここで第2章の原理が具体化される: Rulesが「意図を伝え」（教育的機能）、Hooksが「条件を満たさなければ止める」（検査的機能）。両方が噛み合って初めて、「意味を理解した上で、条件を確実に満たす」成果物が生まれる。

7.6 Step 5 — MCPで外部データを接続する

ここまでは input/ にあるファイルだけを使っていた。MCPで外部データソースを接続する。

たとえば、Filesystem MCPを使えば、プロジェクトディレクトリ外のファイルにもアクセスできる。チームメンバーがそれぞれの作業ディレクトリにメモを置いている場合、MCPを通じてそれらを横断的に読み込める。

何が変わるか: AIが「手足」を得る。コンテキストウィンドウに最初から入っている情報だけでなく、必要に応じて外部から情報を取得できるようになる。

確率的/決定論的の二面性: AIがMCPツールを「いつ使うか」の判断は確率的だ。しかし、ツールが呼ばれた後のファイル読み取りは決定論的に実行される。

7.7 Step 6 — Agentsでタスクを分割する

メモの数が増えてきた場合——たとえば、20人のチームメンバーから毎週メモが来る場合——を考える。一つのコンテキストウィンドウで全メモを処理しようとすると、コンテキストが逼迫する。

Agentsを使って、Leadが統括しWorkerがグループごとに分担処理する構成にできる。

Lead: 全メモを5人ずつ4グループに分配
  ├── Worker A: グループ1のメモを分類・要約
  ├── Worker B: グループ2のメモを分類・要約
  ├── Worker C: グループ3のメモを分類・要約
  └── Worker D: グループ4のメモを分類・要約
Lead: 4つの要約を統合して最終レポートを生成

何が変わるか: コンテキスト分離により、各Workerは少量のメモに集中できる。並列実行で処理時間も短縮される。

何がまだ変わらないか: Worker間の文脈共有にはファイルI/Oが必要で、オーバーヘッドがある。少量のメモならAgentsなしの方がシンプルで効率的だ。

7.8 段階的構築から見える構造

6つのStepを振り返ると、制御の追加は以下のパターンを描いている。

Step 1  CLAUDE.md     → 全体像の把握。概ねうまくいくが揺らぐ。
Step 2  + Rules       → 行動規律の追加。精度は上がるが保証なし。
Step 3  + Skills      → 手順の標準化。再現性が向上するが100%でない。
Step 4  + Hooks       → 決定論的制御の導入。条件の遵守が保証される。
Step 5  + MCP         → 外部世界への接続。情報取得の範囲が広がる。
Step 6  + Agents      → タスクの分割。スケーラビリティが得られる。

最初の3ステップは確率的制御の世界だ。精度は段階的に上がるが、どこまで行っても100%にはならない。Step 4で初めて決定論的制御が導入され、質的な変化が起きる。Step 5〜6は能力の拡張であり、制御の種類としては確率的要素と決定論的要素の複合だ。

重要なのは、すべてのStepが必要なわけではないということだ。週次レポートが「だいたい合っていればよい」なら、Step 1〜2で十分かもしれない。手順の再現性が重要なら、Step 3〜4まで進める。大量データを扱うなら、Step 5〜6が必要になる。タスクの性質に応じて「どこまで馬具を装着するか」を判断する——これが第5章で述べた「自由度の配置」の具体化だ。

実用上のポイント: 新しいタスクを始めるとき、最初からStep 6まで構築しようとしない。Step 1（CLAUDE.mdだけ）から始めて、実際に問題が起きたら次のStepを追加する。「手順が飛ばされた」→ Skillsを追加。「それでも飛ばされる」→ Hooksを追加。問題駆動で段階的に馬具を足していく方が、最初から全部揃えるより効率的で、かつ過剰設計を避けられる。
