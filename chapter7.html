第7章 実装リファレンス

本章は技術者向け。 第6章で述べた各機能の具体的なファイル構成、記述方法、コード例をまとめている。コードを書かない読者は読み飛ばしてよい。

7.1 CLAUDE.md の書き方

プロジェクトルートに CLAUDE.md を配置する。セッション開始時に自動読み込みされる。

# プロジェクト概要

レシピデータを構造化してネットワーク可視化するプロジェクト。

## ディレクトリ構成
- docs/     — 仕様書、進捗
- data/     — 入力データ（生レシピテキスト）
- output/   — 生成されたYAMLとHTML
- scripts/  — 検証・変換スクリプト

## 作業の前提
- 仕様変更は必ず docs/spec.md に記録する
- data/ 内のファイルは読み取り専用として扱う

7.2 Rules の配置と記述

.claude/rules/ ディレクトリにMarkdownファイルを配置する。複数ファイルに分割可能。

# ファイル: .claude/rules/workflow.md

## 実行原則
- 作業開始前に必ず docs/progress.md を読み、現在の状態を確認する
- 1つのPhaseが完了するたびに progress.md を更新する

## 禁止事項
- spec.md に記載されていない機能を独自判断で追加しない
- 既存のテストを削除しない
- 作業対象外のディレクトリを変更しない

7.3 Skills のディレクトリ構成

.claude/skills/スキル名/SKILL.md として配置する。YAML frontmatterで名前と説明を定義。

.claude/skills/
  setup/
    SKILL.md          ← メインの指示（必須）
    templates/        ← テンプレートファイル
  execute/
    SKILL.md
    scripts/
      validate.sh     ← Skillから呼び出せるスクリプト
# ファイル: .claude/skills/report/SKILL.md

---
name: report
description: 週次メモから構造化されたレポートを生成する。
---

## 手順
1. input/ の週次メモファイルを読む
2. メモの内容をカテゴリに分類する
3. サマリーを作成する（3行以内）
4. output/report_YYYY-MM-DD.md としてレポートを出力する
5. 「レポート生成完了」と報告する

旧Commandsとの違い: 旧来のSlash Commands（.claude/commands/setup.md）は単一ファイルでユーザーの手動呼び出しのみだったが、Skillsはディレクトリ構成でスクリプトをバンドルでき、自動発動にも対応する。旧Commandsも後方互換で動作するが、同名の場合はSkillが優先される。

7.4 Hooks の設定

.claude/settings.json でHookを定義する。スクリプトは .claude/hooks/ に配置。

Stop Hook（タスク完了前の検証）:

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "command": "bash .claude/hooks/check_completion.sh"
      }
    ]
  }
}
# ファイル: .claude/hooks/check_completion.sh
#!/bin/bash

errors=""

# progress.md が更新されているか確認
if ! git diff --name-only HEAD | grep -q "progress.md"; then
  errors="${errors}progress.md が更新されていません。\n"
fi

# git commit がされているか確認
if [ -n "$(git status --porcelain)" ]; then
  errors="${errors}コミットされていない変更があります。\n"
fi

if [ -n "$errors" ]; then
  echo "以下の問題があります："
  echo -e "$errors"
  exit 1  # 完了を拒否 → モデルに差し戻し
fi

exit 0  # 完了を許可

PreToolUse Hook（危険な操作の防止）:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "bash .claude/hooks/check_write_permission.sh \"$TOOL_INPUT\""
      }
    ]
  }
}

スクリプトの exit 1 でAIの動作が差し戻される。モデルの外側で動くプログラムなので、AIがこのチェックをスキップする手段はない。

7.5 MCP の接続

MCPサーバーを設定すると、AIが外部リソースにアクセスできるようになる。

モデル ←→ MCP ←→ 外部リソース

具体的な接続例：
  MCP Server: Filesystem  → ローカルファイルの読み書き
  MCP Server: GitHub      → リポジトリ操作
  MCP Server: Slack       → メッセージの送受信
  MCP Server: Database    → SQLクエリの実行

AIが「どのMCPツールをいつ呼ぶか」の判断は確率的だが、ツール呼び出し後の実際の処理は決定論的なコードとして動作する。
