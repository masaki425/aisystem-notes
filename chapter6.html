第6章 Harnessの機能 — 概念と使い分け

第1章〜第5章で述べた原理を、実際のツールではどう実装しているのか。本章ではClaude Codeを例に、各機能が何をするもので、いつ使い、何を保証するかを解説する。コードを書かない読者もここまでは読み進めてほしい。具体的なファイル構成やコード例は第7章（実装リファレンス）にまとめてある。

また、「週次メモから構造化レポートを生成する」という簡単なタスクを題材に、機能を1つずつ追加していく実例（examples/ ディレクトリ）を用意している。各機能の説明を読んだら、対応する実例を試してみてほしい。

6.2 CLAUDE.md — プロジェクトの「地図」

制御の種類: 確率的制御

CLAUDE.mdは、AIにプロジェクトの全体像を伝えるファイルだ。セッションが始まると自動的に読み込まれ、「このプロジェクトは何で、どこに何があり、何を前提とするか」をAIに伝える。人間で言えば、新しく参加したメンバーに渡すプロジェクト概要書にあたる。

書くべき内容はシンプルで、プロジェクトの目的、フォルダ構成、作業上の前提条件だ。「data/は読み取り専用」「仕様変更はspec.mdに記録する」のような基本ルールもここに書ける。

限界: CLAUDE.mdはテキストとしてコンテキストに入るだけなので、書いてあることを必ず守る保証はない。「data/は読み取り専用」と書いてあっても、AIがうっかり書き込むことはありうる。重要な制約は、後述のHooksで二重に守る必要がある。

実例 01: examples/01_claude_md_only/

CLAUDE.mdだけの最小構成。「週次レポートを作って」と指示して、構成指示がどこまで守られるかを観察する。同じ指示を2回実行して、結果が同じかどうかも確認してみよう。確率的制御の「揺らぎ」を体感できる。

6.3 Rules — 行動規則の定義

制御の種類: 確率的制御（CLAUDE.mdより強い影響力）

Rulesは、AIの行動規則を定義するファイルだ。CLAUDE.mdがプロジェクト全体の「地図」なら、Rulesは「交通ルール」にあたる。「やるべきこと」「やってはいけないこと」を明確に列挙する。

CLAUDE.mdとの違いは、より具体的で指示的な内容に特化していること。「ファイル名はreport_YYYY-MM-DD.mdとする」「メモに書かれていない情報を捏造しない」「主観的な評価を追加しない」のように、遵守すべきルールを個別に記述する。

限界: Rulesもテキストとしてコンテキストに注入される確率的制御だ。守られることが多いが、AIが「これは改善だ」と独自に解釈してルールを逸脱することがある。これは「ルールを無視した」のではなく、「ルールの解釈が人間と異なった」という確率的処理の結果だ。

実例 02: examples/02_with_rules/

実例01にRulesを追加した構成。実例01と同じ指示で、Rulesの有無で振る舞いがどう変わるかを比較する。Rulesが守られないケースにも注目しよう。

6.4 Skills — 再利用可能なワークフロー定義

制御の種類: 確率的制御

Skillsは、定型作業の手順を再利用可能な形で定義する仕組みだ。毎回「こうやって、ああやって」と手で指示するのではなく、手順を一度定義しておけば /スキル名 で呼び出せる。人間で言えば「業務マニュアル」にあたる。

Skillsの特徴は2つある。第一に、ユーザーが /report のように手動で呼び出せること。第二に、AIがタスクの内容に基づいて自動的に適用する場合があること（たとえば「週報を作って」と言うだけで、reportスキルが自動発動する）。

もう一つの重要な特徴は、指示テキストだけでなくスクリプトやテンプレートをバンドルできることだ。これにより、確率的制御（指示テキスト）と決定論的制御（バンドルされたスクリプト）を一つのパッケージに統合できる。

限界: Skillsの指示テキストに対する遵守は確率的だ。「手順1〜5を順に実行せよ」と書いても、手順3をスキップすることはありうる。手順の確実な実行が必要な場合は、Hooksによる検証が必要になる。

実例 03: examples/03_with_skill/

実例02にSkillを追加した構成。/report と入力するだけで手順とテンプレートが適用される。自由指示（実例02）とSkill呼び出し（実例03）で品質の違いを比較する。

6.5 Hooks — 決定論的制御の実装

制御の種類: 決定論的制御

Hooksは、ここまでの機能とは質的に異なる。CLAUDE.md、Rules、Skillsがすべて「テキストによる影響」だったのに対し、Hooksはプログラムによる強制だ。

仕組みはシンプルで、AIの動作中の特定タイミング（タスク完了時、ファイル書き込み時など）に、検証用のスクリプトを自動実行する。スクリプトが「NG」を返せば、AIの動作は差し戻される。AIにこのチェックを「スキップ」する手段はない。

たとえば、「レポートに必須セクションが含まれているか」「サマリーが3行以内か」「input/が変更されていないか」をスクリプトでチェックする。Rulesにも同じことを書いてあるが、Hooksで二重に守ることで、確率的にスキップされるリスクがゼロになる。

Hooksが解決する本質的な問題: 第2章で述べた「確率的制御の限界」に対する、現時点での最も実用的な解答がHooksだ。AIが「完了です」と言っても、条件を満たしていなければ自動的に差し戻す——この「ゲート」はモデルの外側で動くプログラムなので、確率的な揺らぎに影響されない。

実例 04: examples/04_with_hook/

実例03にStop Hookを追加した構成。レポート生成後、検証スクリプトが自動実行される。実例03では「サマリー3行以内」がスキップされることがあったが、実例04では超過は許されない。これが確率的制御と決定論的制御の違いの実感。

6.6 MCP（Model Context Protocol）— 外部世界への接続

制御の種類: ツール選択は確率的制御、ツール実行は決定論的

MCPは、AIと外部ツール・データソースを接続するための標準プロトコルだ。AI単体はテキストを生成することしかできないが、MCPを通じてファイルの読み書き、データベースの操作、外部APIの呼び出しなど、外部世界に作用できるようになる。人間で言えば「手足」にあたる。

MCPには確率的な側面と決定論的な側面がある。AIが「どのツールをいつ使うか」の判断は確率的だ。しかし、ツールが呼ばれた後の実際の実行（ファイルの読み書きなど）は決定論的なプログラムとして動作する。

6.7 機能の使い分け — 確率的/決定論的の配置

ここまでの機能を、制御の種類で整理する。

確率的制御（Context Engineering）
  ┃
  ┣━ CLAUDE.md    ── 常時読み込み。プロジェクトの前提。
  ┃                  最も広い範囲に影響するが、最も緩い制御。
  ┃
  ┣━ Rules        ── 常時読み込み。行動規則・禁止事項。
  ┃                  CLAUDE.mdより具体的な指示。
  ┃
  ┣━ Skills       ── 呼び出し時 or 自動発動。ワークフロー定義。
  ┃                  手順の一貫性を高める。スクリプトもバンドル可能。
  ┃                  指示テキスト部分はスキップされうる。
  ┃
  ┗━ MCP定義      ── ツールの説明。いつ何を使うかの判断。
                     モデルが使うかどうかは確率的。

決定論的制御（Architectural Constraints）
  ┃
  ┣━ Hooks        ── イベント駆動。条件チェック。
  ┃                  モデルがスキップできない強制力。
  ┃
  ┗━ MCP実行      ── ツール呼び出し後の実際の処理。
                     コードとして確実に実行される。

実践上の設計パターン: 重要な制約は「Rulesに書く＋Hooksでも守る」の二重構造にする。Rulesだけだと確率的に破られるリスクがある。Hooksだけだと、なぜその制約があるのかモデルが理解できず、不自然な動作をする可能性がある。両方を組み合わせることで、モデルは制約の意図を理解しつつ（Rules）、万が一忘れても強制される（Hooks）。

6.8 ハンズオン — 4ステップで体感する二重構造

ここまで読んだら、examples/ ディレクトリの4つの実例を順番に試してほしい。同じタスク（週次メモからレポートを生成）に対して、Harnessの機能を1つずつ追加していく。

examples/
  01_claude_md_only/   ← CLAUDE.md だけ
  02_with_rules/       ← + Rules
  03_with_skill/       ← + Skill（/report で呼び出し）
  04_with_hook/        ← + Hook（Stop hookで品質検証）

実験の手順:

1. 各ディレクトリにClaude Codeで cd する

2. 01〜02では「週次レポートを作って」と指示する

3. 03〜04では /report と入力する

4. 生成されたレポートを比較する

観察すべきポイント:

比較何が変わるか何を学べるか
01→02Rulesの追加でファイル命名規則や禁止事項が反映されるか確率的制御の「影響力」と「不確実性」
02→03Skillの導入で手順とテンプレートが標準化されるか再利用可能なプロンプトの価値
03→04Hookの追加で必須条件が確実に守られるか確率的制御と決定論的制御の質的な違い

最も重要なのは03→04の差だ。03では「サマリー3行以内」がスキップされることがあるが、04ではbashスクリプトが必ずチェックするため、超過は差し戻される。この差が、本稿全体を通じて述べてきた「確率的制御の限界」と「決定論的制御の必要性」の実感になる。

6.9 実証事例 — Hooksの有無が結果を決定的に変える

以下は、同じワークフローを3つの異なるツールで実行した比較実験の結果である。

指示内容Claude Code（Hooks有）Codex CLI（Hooks無）Gemini CLI（Hooks有）
Phase毎にgit commit全て実行された全てスキップされた実行された
progress.md の更新毎回更新されたまとめて事後記録更新された
手順の順序遵守順番通り実行一括実行・順序無視順番通り実行

Codex CLIではHooksに相当する機構がないため、プロンプトに「Phase毎にcommitせよ」と何度書いても、全てスキップされた。これはプロンプトの書き方の問題ではなく、アーキテクチャの問題だ。

手順保証はプロンプトの工夫では達成できない。決定論的制御（Hooks）が必要。

これは書き方の問題ではなく、LLMのアーキテクチャ的な限界である。

6.10 ワークフローの3層構造

実験を通じて、AIワークフローは以下の3層からなることが明らかになった。

第1層: ドキュメント層 → ツール間で完全に共通
         spec.md, proposal.md, progress.md, issues.md
         → 外部永続化層に対応

第2層: 実行環境層   → 軽微な変換で移植可能
         Skills, Rules, エージェント定義
         → Context Engineering層に対応

第3層: 手順保証層   → 移植不可能（Hooks依存）
         Hooks（Stop, PreToolUse）
         → Architectural Constraints層に対応

第1層と第2層は確率的制御の世界にあり、ツール間で共有・移植できる。第3層は決定論的制御であり、その仕組みを持たないツールには移植できない。

6.11 実践 — Claude Codeでの開発プロセス

ここまでの機能が実際の開発でどう組み合わさるかを、一連の流れとして示す。

#### Phase 0: 構想を対話で練る（Claude.ai）

まず、チャットUIで構想を対話的に練り上げる。ここではHooksもSkillsも使わない。自由度の高い環境で、第3章で述べた「思考をぶつける」を実践する。

人間: 「レシピデータを概念ネットワークとして可視化したい。
       手元にレシピテキストが50件ある。D3.jsで可視化する想定。」

AI:   「ノードの種類（食材/調理法/味）で3層構造になる。
       エッジの定義が重要で...5基準スコアリングがある...」

人間: 「50件で5基準だと組み合わせ爆発しないか？
       チャンク分割して処理すべきでは？」

AI:   「チャンク分割は有効。ただし重複ノード処理が課題。
       chunk_{N}_プレフィックスで管理して後でマージする...」

#### Phase 1: proposal.mdの作成（Claude.ai）

対話で練った構想を、proposal.md（提案書）として構造化する。第3章で述べたインプットの4要素（目的・文脈・仮説・判断基準）を文書化する作業だ。

#### Phase 2: セットアップ（Claude Code）

Claude Codeに移り、proposal.mdに基づいてHarness一式（CLAUDE.md、Rules、Skills、Hooks）を生成する。馬に馬具を装着する作業。

#### Phase 3: 実行（Claude Code）

/execute でSkillを呼び出し、実装を開始する。Hooksが裏側で動き、各Phase完了時に品質を検証する。人間は見守るだけ。

#### Phase 4: レビューと対話

成果物を確認し、「思考をぶつける」対話に戻る。修正が必要であればspec.mdを更新し、再度実行する。

#### 開発プロセスの全体像

Phase 0: 構想（対話）        自由度：最大 ／ 制御：確率的のみ
  ▼
Phase 1: 提案書（対話+構造化） 自由度：高   ／ 制御：確率的のみ
  ▼
Phase 2: セットアップ         自由度：低   ／ 制御：確率的 + 決定論的
  ▼
Phase 3: 実装（自律実行）     自由度：最低 ／ 制御：確率的 + 決定論的
  ▼
Phase 4: 評価（対話）        自由度：最大 ／ 制御：確率的のみ
  ├── 満足 → 完了
  └── 修正要 → Phase 1 or 3 へ戻る

このプロセス全体を貫いているのは自由度のグラデーションだ。最初は自由度が最大で、セットアップで徐々に固まり、実装では最小になり、評価でまた最大に戻る。どのフェーズでも、人間の仕事は「思考をぶつける」ことだ。
