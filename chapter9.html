第9章 Harnessの移植 — 別のAIツールで同じHarnessを動かす

本章の主題。 同じ目的のHarnessを別のLLMツールに移植するとき、何がそのまま動き、何を書き換える必要があるのか。Claude Code で構築したHarnessを Codex CLI と Gemini CLI に移植した実験結果に基づいて述べる。

9.1 モデル層の移植 — 自然言語がコードである

プログラミング言語の世界では、Java で書いたコードを Python に移植するには書き直しが必要だ。言語仕様が異なるため、同じロジックを別の構文で表現し直さなければならない。

LLM ベースのシステムでは、指示が自然言語で書かれている。自然言語は「プログラミング言語」に相当するが、決定的に異なる点がある——全てのモデルが同じ言語を読める。日本語で書かれた spec.md は、Claude にも GPT にも Gemini にも渡せる。構文変換は不要だ。

従来のプログラミング：
  Java → Python 移植 = 書き直し

LLM ハーネス：
  自然言語の指示 → 別のモデルに渡す = そのまま読める

これは大きな利点だ。仕様書（spec.md）、進捗記録（progress.md）、提案書（proposal.md）、課題管理（issues.md）——これらのドキュメント層は、ツールを問わず完全にポータブルである。

9.2 同じ文章、異なる解釈

しかし、「読める」ことと「同じように解釈する」ことは別の問題だ。

人間でも同じ文章を読んで異なる理解に至ることがある。LLM でも同様に、各モデルには訓練データ・学習方法・アーキテクチャの違いに起因する「個性」がある。同じプロンプトを渡しても、出力が異なる。

実験で確認された個性の違いを示す。同一の proposal.md を3つのツールに渡して「必要なファイルを全部生成せよ」と指示した結果は以下の通りだった。

観点Claude CodeCodex CLIGemini CLI
指示の解釈ワークフロー（作業の流れ）管理ファイル生成実装コード生成＋実行実装コード生成＋実行
生成物spec.md / progress.md / agents/src/ + run_pipeline.pysrc/ + main.py
Phase分離あり（A→B→C 逐次）なし（直接実行）なし（直接実行）

Claude Code だけがワークフロー（作業の流れ）管理ファイルを先に生成した。これは Claude の「個性」のように見えるが、実際にはそうではない。Claude Code のプロジェクトには /setup コマンドファイル（.claude/commands/setup.md）が事前に配置されており、Claude はそれを読んで「まず管理ファイルを作る」という行動を取った。一方、Codex と Gemini にはそのようなコマンドファイルの仕組みがなかったため、proposal.md を「コーディング指示」として直接解釈し、実装に走った。

つまり、この差は「モデルの個性」ではなく「Harness の有無」に起因する。Harness がなければ、モデルは自分の個性（事前学習のバイアス）に従って自由に解釈する。Harness があれば、解釈の方向を制御できる。

ただし、Harness を整えた上でも残る個性の違いは確かに存在する。同じ構造化タスクを与えた結果、生成されたノード数とエッジ数に大きな差が出た。

指標Codex CLIGemini CLI
ノード数58約15
エッジ数4610
5基準スコア付与率100%（テンプレート値）40%（6/10エッジで欠落）
クロスレシピエッジほぼなしゼロ

Codex は「網羅的に生成するが、スコアを個別に評価せずテンプレート値で埋める」という傾向を示した。Gemini は「過度に凝縮し、指示された項目を欠落させる」という傾向を示した。いずれも、同じ自然言語の指示書を読んだ上での結果だ。

9.3 実行環境層の移植 — リネームとフォーマット変換

モデル層（ドキュメント）は移植不要だが、Harnessの実行環境層は書き換えが必要になる。各ツールが独自の仕様を持つからだ。

指示ファイルの名前が異なる。Claude Code は CLAUDE.md、Codex CLI は AGENTS.md、Gemini CLI は GEMINI.md を自動読み込みする。名前が違うだけで中身は同じ自然言語なので、リネームすれば済む。

コマンド体系のフォーマットが異なる。Claude Code の Slash Commands（.claude/commands/setup.md）は独自形式だが、Codex CLI と Gemini CLI は共通の SKILL.md オープン標準を採用している。変換には YAML frontmatter の追加が必要になるが、指示本体は同じ自然言語なのでそのまま流用できる。

Claude Code:
  .claude/commands/setup.md      ← 独自形式
  .claude/commands/execute.md

Codex CLI:
  .agents/skills/setup/SKILL.md  ← SKILL.md 標準
  .agents/skills/execute/SKILL.md

Gemini CLI:
  .gemini/skills/setup/SKILL.md  ← 同じ SKILL.md 標準
  .gemini/skills/execute/SKILL.md

Codex と Gemini が同一の SKILL.md 標準を使っているため、Claude Code → SKILL.md への変換を一度行えば、2ツール分の対応が完了する。

ルールの配置場所が異なる。Claude Code は .claude/rules/ ディレクトリに独立ファイルとして配置するが、Codex CLI は AGENTS.md に統合転記し、Gemini CLI は GEMINI.md に統合転記する。ルールの内容自体は同じ自然言語なので、テキストのコピー＆ペーストで済む。

Worker定義とエージェント起動方式が異なる。Worker定義ファイルの中身は全ツールで共通の自然言語だが、配置パスが異なる（Claude Code は .claude/agents/、Codex/Gemini は docs/agents/）。より本質的な差は起動方式にある。Claude Code ではTaskツールで最大10のWorkerを並列起動できる。Codex CLI にはネイティブの並列実行機構がなく、逐次実行に書き換える必要がある。Gemini CLI はSubagents（実験的機能）で部分的に対応している。並列実行を逐次に変える書き換えは1箇所で済むが、処理時間には影響が出る。

これらの変換作業をまとめると、大部分はリネームとパス変更であり、実質的な書き換えは各ツール数箇所に収まる。

9.4 手順保証層の移植 — 機能の非対称性

ここまでの移植は比較的容易だった。しかし、Hooks（手順保証層）の移植は根本的に異なる問題を提起する。

第2章で述べたように、Hooks は決定論的制御であり、モデルがスキップできない強制機構だ。この機構の実装は各ツールで大きく異なる。

機能Claude CodeCodex CLIGemini CLI
Stop hook（終了前チェック）ネイティブ対応なしネイティブ対応
PreToolUse hookネイティブ対応なしネイティブ対応
Hook管理settings.json—/hooks list/enable/disable
代替手段—git pre-commit hook—
LLM行動の強制可能不可能可能

Claude Code と Gemini CLI は Hooks をネイティブに持っているため、「progress.md を更新するまで完了を許可しない」という制約を確実に強制できる。

Codex CLI にはこの機構がない。git pre-commit hook で部分的に代替できるが、それは「commit 時のチェック」であって「LLM の行動をリアルタイムに制御する」ものではない。根本的な設計の違いであり、プロンプトの工夫では補えない。

9.5 手順保証が欠如するとどうなるか

Hooksを持たないツールで同じHarnessを実行すると何が起きるか。Codex CLIでの実験結果を示す。

git commitが一度も実行されなかった。 Skillsに「Phase完了ごとにcommitせよ」と繰り返し記述していたが、全てスキップされた。

ステップ順序が無視された。 ログの全エントリのタイムスタンプが同一だった。Codexはパイプライン全体を一括実行し、ログを事後的にまとめて書いたと考えられる。

原因はプロンプトの書き方ではない。 Codexはプロンプトを「従うべき手順」ではなく「最終目標の記述」として解釈する。「成果物を作る」に直接関係しないステップ（commit、ログ記録）は自己判断で省略する。指示の書き方を変えても同じ結果になった。

これは第2章の原則——「確率的制御は遵守率100%にならない」——の具体的な実証例だ。自然言語で「必ずcommitせよ」と書いても、モデルにとっては「提案」にすぎない。Hooksという決定論的な強制機構がなければ、手順遵守は保証できない。

9.6 三層ポータビリティ・モデル

以上の移植実験から、Harness は以下の3層に分けて移植可能性を評価できることがわかった。

第1層: ドキュメント層  → 完全にポータブル ✅
       proposal.md, spec.md, progress.md, issues.md, agent_log.md
       自然言語で書かれているため、全ツールでそのまま使える。

第2層: 実行環境層  → 軽微な変換で対応可能 ✅
       commands/skills, rules, agents 定義, Python スクリプト
       フォーマット変換（リネーム、YAML frontmatter 追加、パス変更）で対応。
       各ツール3箇所程度の実質的な書き換えで済む。

第3層: 手順保証層  → ツール依存、移植不可能な場合がある ❌
       Hooks（Stop, PreToolUse）による手順遵守の強制。
       Claude Code と Gemini CLI はネイティブ対応。
       Codex CLI には同等の機構がなく、代替も限定的。

この三層モデルは、移植の際の判断基準になる。第1層と第2層が共通なので、ツール間の「引っ越し」は比較的容易だ。問題になるのは第3層だけであり、それは「手順の遵守がどれだけ重要か」というプロジェクトの性質によって判断が分かれる。

9.7 移植の実用的な指針

三層モデルに基づく実用的な指針を示す。

ドキュメント層は最初からポータブルに設計する。 proposal.md や spec.md にツール固有の記法を混ぜない。自然言語で完結させる。これは「将来の移植」のためだけでなく、「人間が読んで理解できる」ためでもある。

実行環境層は変換スクリプトで対処する。 Claude Code → Codex/Gemini への変換は定型的なので、シェルスクリプトで自動化できる。実際に convert_to_codex.sh として実装・運用している。

手順保証層はツール選択で対処する。 Hooks が必要なプロジェクトでは、Hooks をネイティブに持つツール（Claude Code、Gemini CLI）を選ぶ。成果物さえできればよいプロジェクトでは、どのツールでも動く。

用途別の推奨ツール：

手順遵守が重要 → Claude Code（Hooks で強制できる）
成果物重視     → どれでも可（ドキュメント層はポータブル）
外部レビュー   → Codex / Gemini（異なるモデルの目が必要）
大規模入力     → Codex / Gemini（1M tokens 対応）

9.8 移植が明らかにしたこと

移植実験は、Harness の各部分の役割を改めて明確にした。

ドキュメント層がポータブルなのは、自然言語が「万国共通」だからだ。どのモデルも日本語を読める。これはプログラミング言語の移植には存在しない利点であり、LLM ベースのシステム設計における本質的な強みだ。

実行環境層の変換が軽微なのは、各ツールが「同じことを異なるフォーマットでやっている」からだ。コマンドの呼び出し方やルールの配置場所は異なるが、機能的にはほぼ等価だ。

手順保証層が移植困難なのは、これが唯一「コード実行」に依存する部分だからだ。自然言語による指示（確率的制御）は全ツール共通だが、コードによる強制（決定論的制御）はツールのアーキテクチャに依存する。

まとめ: 自然言語で書かれた部分は移植が容易で、コードで実装された部分は移植が困難だ。これは直感に反するように見えるが、LLM の世界では自然言語こそが共通インターフェースであり、ツール固有のコード実行基盤こそが「方言」にあたる。

実用上のポイント: ツール選択で最初に問うべきは「手順の遵守がどのくらい重要か」だ。重要なら、Hooksをネイティブに持つツールを選ぶ。成果物さえできればよいなら、ドキュメント層はどのツールでもそのまま動くので、好みや他の条件（コンテキストサイズ、コストなど）で選んで構わない。ツールに合わせてHarnessを設計するのではなく、Harnessの要件に合わせてツールを選ぶ。
