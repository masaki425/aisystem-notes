# Cycle 3 — proposal.md v1.2 修正と再実行

## サイクル2で発見された問題

### 問題1: 統合ネットワークの接続密度が低い

60ノード中47ノード（78%）が他論文と一切接続していない。統合ネットワークと言いつつ、実態はGianni・Moody・Yarusの3つの島が3個の共有ノード（rna_world, in_vitro_selection, prebiotic_chemistry）で薄くつながっているだけ。

論文間エッジは11本あるが、60ノード・68エッジの規模に対して不十分。3本の論文が「生命の起源」という共通テーマを持つ割には、ネットワークとしての統合が弱い。

### 問題2: Moody論文がrna_worldノードを使用しなかった

LUCAの論文がRNAワールドに言及しないのは不自然。seed listでは `rna_world` の該当論文を「Gianni, Yarus（Moodyでも言及があれば使用）」と書いていた。Worker_moodyは「言及があれば」を「必須ではない」と解釈してスキップした。

同様に `genetic_code` も「Yarus（Moodyでも言及があれば使用）」としていたが、Moody論文（LUCAのゲノム再構成）が遺伝暗号に言及しないはずがない。

### 8.1のサイクル図に照らした判断

**問題1は成功基準レベル**。B3の収束条件「論文間エッジが1本以上」が甘すぎた。基準自体の改善が必要。サイクル1→2が「仕様の不備を修正する」だったのに対し、サイクル2→3は「成功基準自体を精緻化する」——異なるレイヤーの修正。

**問題2は仕様レベル**。seed listの「言及があれば使用」という曖昧な表現が確率的制御の限界を露呈した。明示的な要求に切り替える。

→ **B3（収束条件）とB5（seed list運用ルール）の両方を修正する**

## proposal.md v1.1 → v1.2 の変更内容

### 変更箇所1: B3（収束条件）の強化

**変更前**:
```
- 条件3: 統合YAMLに論文間をつなぐエッジが1本以上存在すること
```

**変更後**:
```
- 条件3: 統合YAMLにおいて、各論文ペア間（G-M, G-Y, M-Y）にそれぞれ2本以上の論文間エッジが存在すること
- 条件4: 統合後に他論文と接続のないノードが全体の60%以下であること
```

**変更理由**: 「1本以上」ではどの論文ペア間にエッジがあるかすら保証されない。サイクル2ではG-M間、G-Y間、M-Y間にエッジがあったが、その分布は不均一で、78%のノードが孤立島状態だった。論文ペアごとの最低本数と、孤立ノード率の上限を設定する。

**設計判断**: 60%という閾値は、サイクル2の78%から明確に改善されたことを判定できる程度の値。完璧を求めると（例: 30%以下）、強引なエッジ追加を誘発するリスクがある。satisficingの原則に従い、「明確に改善された」と判定できる閾値を設定する。

### 変更箇所2: B5（seed list）の該当論文を明示的要求に変更

**変更前**:
```
| `rna_world` | RNAワールド | concept | Gianni, Yarus（Moodyでも言及があれば使用） |
| `genetic_code` | 遺伝暗号 | concept | Yarus（Moodyでも言及があれば使用） |
```

**変更後**:
```
| `rna_world` | RNAワールド | concept | Gianni, Moody, Yarus |
| `genetic_code` | 遺伝暗号 | concept | Yarus, Moody |
```

**変更理由**: 「言及があれば使用」という表現は確率的制御として曖昧すぎた。Workerは「必須ではない」と解釈してスキップした。LUCA論文がRNAワールドと遺伝暗号に言及しないのはドメイン知識として不自然であり、該当論文のリストに明示的に含めることで対処する。

### Stop Hookの発動事例 — 決定論的制御と確率的制御の対比

サイクル3の実行中、Stop Hookが1回発動した:

```
Stop hook error: Prompt hook condition was not met:
Worker agents are still running (worker_moody pending).
Phase 4 (merge.py + 論文間エッジ追加) and final validation
must complete before git commit.
Fix: Wait for worker_moody to complete, then execute
Phase 4 merge, validation, and final git commit.
```

worker_gianniとworker_yarusが先に完了し、worker_moodyがまだ処理中のときに、LeadがPhase 4（マージ）に進もうとした。Stop Hookがこれを検出し、worker_moodyの完了を待つよう差し戻した。

**これは決定論的制御が正しく機能した事例**。Hookに入れたもの（progress.md更新 + git commit）は、副産物としてWorker未完了も検出できた。

**同じサイクルで、B3条件4（孤立ノード率）は確率的制御の甘い評価が通った**。Hookに入れなかったものは、Leadが「全項目PASS」と報告した。

| 制御の種類 | 対象 | 結果 |
|----------|------|------|
| 決定論的（Hook）| Worker未完了の検出 | ✅ 止まった。差し戻された |
| 確率的（Leadの判断）| B3条件4の孤立ノード率 | ❌ 甘く評価されて通った |

これが第2章で述べた「確率的制御と決定論的制御の使い分け」の生きた教材。同じサイクルの中で、Hookに入れたものと入れなかったものの差が明確に出た。

### 観測的見立て — なぜLeadはB3条件4を正しく判定できなかったか

B3条件4の甘い評価の原因は、数理的背景の不足にあると推測される。問題は2層ある。

**第1層: 計算手順が与えられていない**。B3条件4は「他論文と接続のないノードが全体の60%以下」と数値で定義されている。しかし、この数値を算出する具体的な手順——「各ノードについて、そのIDが他論文のノードIDと共有されているか、またはエッジのsource/targetとして他論文のノードと接続しているかを判定し、接続なしのノード数を全ノード数で割る」——がspec.mdにもWorker定義にも書かれていない。Leadは「なんとなく接続が増えた」と感覚的に判断してPASSにした可能性が高い。

**第2層: 数値基準と確率的評価の本質的な不整合**。60%という閾値は数値なのに、検証方法が確率的（Leadの判断）。数値基準を設けたなら、計算で判定すべき。条件3（各ペア間に2本以上）は数が少ないのでLeadが正しく数えられたが、条件4は60ノード全体の接続状況を評価する必要があり、LLMが暗算で正確に判定するのは無理がある。

**教訓**: 数値で表現された基準は、スクリプト（決定論的制御）で検証すべき。確率的制御に数値判定を委ねると、「なんとなくOK」が通る。E4で「Hookで強制→過剰」と判断したが、数値基準については「過剰」ではなく「必要」だった可能性がある。これは8.1の「自由度の配置」の判断を誤った具体例として、8.6本文で取り上げる価値がある。

## Satisficing 判断

### A3 成功基準との照合

- 基準1: 3本すべての論文が個別にYAML化 ✅
- 基準2: 個別YAMLがスキーマ準拠 ✅
- 基準3: 統合YAMLが生成・スキーマ準拠 ✅
- 基準4: 統合YAMLに3本すべてのDOI含む ✅

### B3 収束条件との照合

- 条件1: 各論文15ノード以上 ✅（20, 23, 22）
- 条件2: 各論文15エッジ以上 ✅（21, 25, 21）
- 条件3: 各論文ペア間に2本以上のエッジ ✅（G-M:8, G-Y:8, M-Y:9）
- 条件4: 孤立ノード率≤60% ❌（70%）

### B3条件4の未達に対する判断

孤立ノード率は78%→70%に改善したが、60%の閾値には未達。ここでサイクル4に進むか、satisficeするかの判断が必要。

**サイクル3で完了とする理由**:
1. 改善の方向は正しい（78%→70%）。問題は解決に向かっている
2. さらに孤立率を下げるには、論文固有の専門的な概念間にエッジを張る必要があるが、ドメイン知識なしに強引にエッジを追加すると品質が下がる
3. 60%という閾値自体がサイクル2の78%から「明確に改善された」ことを判定するために設定した値であり、絶対的な基準ではない
4. デモの焦点は「反復サイクルによるHarness改善」であり、ネットワークの完璧さではない
5. 8.5（satisficing）の原則: 完璧を追求するのではなく、十分に良い解を受け入れる

**B3条件4をPASSに修正するのではなく、「未達だがsatisficeする」と判断する**。これ自体がsatisficingの実例。

### サイクルごとの「戻り先」の違い

| サイクル | 問題の性質 | 8.1の戻り先 |
|---------|----------|------------|
| 1→2 | 仕様の不備（命名規則がない）| 仕様レベル（proposal.md B5）|
| 2→3 | 成功基準の甘さ + 仕様の曖昧さ | 成功基準レベル（B3）+ 仕様レベル（B5運用ルール）|
| 3→完了 | 条件4未達だが改善中 | サイクルの出口（satisficing）|

各サイクルで「どこに戻るか」の判断が異なる。これが8.1で述べた「問題の粒度に応じた自由度に戻る」の実践。
