# Cycle 4 — validate_phase.pyにB3条件4のチェックを追加

## サイクル3で発見された問題

### Leadの自己評価が甘い — B3条件4が未達なのに「全項目PASS」と報告

サイクル3の実行結果で、孤立ノード率は70%だった（閾値60%）。しかしLeadは「収束条件: 全4項目PASS」と報告し、独立検証で初めて未達が判明した。

同じサイクルでStop Hookは正しく機能した（worker_moody未完了のままPhase 4に進もうとしたのを差し戻した）。Hookに入れたものは止まり、入れなかったものは甘い評価が通った。

### 問題の根本原因（観測的見立て）

**第1層**: B3条件4の計算手順がspec.mdにもWorker定義にも書かれていない。Leadは「なんとなく接続が増えた」と感覚的に判断した。

**第2層**: 数値基準と確率的評価の本質的な不整合。60%という閾値は数値なのに、検証方法が確率的（Leadの判断）。60ノード全体の接続状況をLLMが暗算で正確に判定するのは無理がある。

### 8.1のサイクル図に照らした判断

これは**実装レベル**の修正。proposal.mdの意図（B3条件4）は正しいが、それを検証する仕組み（validate_phase.py）が不足している。

サイクル1→2: 仕様の不備 → proposal.md修正
サイクル2→3: 成功基準の甘さ → proposal.md修正
**サイクル3→4: 検証の不備 → validate_phase.py修正**

「どこに戻るか」が毎回異なる。今回はproposal.mdではなくスクリプトに戻る。

### E4の設計判断の修正

サイクル3でE4に「Hookで強制→過剰。仕様の明確化で十分と判断」と書いた。しかし結果として数値基準の確率的評価は不十分だった。

修正後の判断: **定性的な基準**（「概念が妥当か」等）は確率的制御で十分。**定量的な基準**（「60%以下か」等）は決定論的制御（スクリプト）で検証すべき。自由度の配置の判断を「定性的か定量的か」で分ける。

## 変更内容

### 変更箇所1: validate_phase.py — validate_final()にB3条件3・4のチェックを追加

**追加するチェック**:

- **条件3**: 各論文ペア間（G-M, G-Y, M-Y）にそれぞれ2本以上の論文間エッジが存在すること
- **条件4**: 統合後に他論文と接続のないノードが全体の60%以下であること

**算出手順（条件4）**:
1. 3本の個別YAMLからノードIDのセットを取得（G, M, Y）
2. merged.yamlの各エッジについて、sourceとtargetがどの論文に属するか判定
3. source論文 ≠ target論文のエッジに接続されているノードを「他論文接続あり」とカウント
4. 共有ノード（2論文以上に属するID）も「他論文接続あり」に含める
5. 「他論文接続なし」のノード数 / 全ノード数 を算出
6. 60%を超えていれば❌

### 変更箇所2: proposal.md D5 — 検証方法の更新

**追加内容**: 第2段階の統合YAML検証に以下を追加:
- 11. 各論文ペア間に2本以上の論文間エッジが存在すること
- 12. 他論文と接続のないノードが全体の60%以下であること

**変更理由**: サイクル3でLeadの確率的評価が甘く、数値基準が正しく検証されなかった。数値基準はスクリプトで検証すべき。

### 変更箇所3: proposal.md E4 — 設計判断の修正

**追加内容**:
| 数値基準をスクリプト化 | サイクル3でLeadが条件4を甘く評価。数値基準は確率的制御では不十分 | 全ての収束条件をHook化（→ 定性的基準には過剰。定量的基準のみスクリプト化） |

### 変更箇所4: proposal.md 変更履歴

v1.3: D5の検証方法にB3条件3・4のスクリプト検証を追加（サイクル3でLeadの数値判定が甘かったため）
