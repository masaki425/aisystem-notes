# Step 4: Hooksを追加して構造化を実行

## 目的

Step 3のCLAUDE.md + Rules + Skillsに加えて、Hooks（Stop hook + 検証スクリプト）を追加し、決定論的制御によって出力の整合性が保証されるかを観察する。

## 追加したファイル

### scripts/validate_yaml.sh

シェルスクリプトによる9項目の整合性チェック:
1. output/ にYAMLファイルが存在するか
2. nodesセクションが存在するか
3. edgesセクションが存在するか
4. ノードが1個以上あるか
5. エッジが1個以上あるか
6. **エッジのsource/targetがすべてnodesのIDに存在するか**
7. **孤立ノード（どのエッジからも参照されないノード）がないか**
8. typeが許可リスト7種に含まれるか
9. relationが許可リスト8種に含まれるか

チェックに一つでも失敗すると `BLOCKED` を出力し、exit 1 で終了する。

### .claude/settings.json

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hook": {
          "type": "command",
          "command": "bash scripts/validate_yaml.sh"
        }
      }
    ]
  }
}
```

Stop hookとして設定。AIが「完了」と判断したタイミングでvalidate_yaml.shが自動実行される。検証に失敗すれば完了が許可されず、AIは問題を修正して再度完了を試みる。

### Claude Codeへの指示

```
/clear してからセッション再開
chmod +x scripts/validate_yaml.sh
input/の論文を構造化してYAMLにして
```

## Hooksの効果（Step 3との比較）

### 最も重要な変化: 孤立ノード問題の解決

Step 2で発見され、Step 3ではSkillの整合性チェック指示にもかかわらず修正されなかった孤立ノード `ribozyme_abundance` が、Step 4では修正された。

**Step 2・Step 3の状態:**
- `ribozyme_abundance` はnodesに存在するが、edgesのどのsource/targetにも参照されていない
- Skill Step 4 に「孤立ノードがあれば削除するかエッジを追加する」と明記していたが、実行されなかった

**Step 4の状態:**
- 新しいエッジが追加された:
  ```yaml
  - source: qt45_ribozyme
    target: ribozyme_abundance
    relation: supports
    description: QT45の発見がRNA配列空間におけるポリメラーゼモチーフの存在頻度が従来想定より高いことを示唆
  ```
- validate_yaml.sh の孤立ノード検出がこれを強制した

**これが決定論的制御の本質的な違い:**
- Skills（確率的制御）: 「確認してね」→ 無視されうる
- Hooks（決定論的制御）: 「確認できるまで完了させない」→ 必ず修正される

### 概要把握セクションの出現

Step 3ではSkillに「概要把握をYAMLコメントに記録する」と指示したが出力されなかった。Step 4ではYAMLの冒頭にコメントとして記録された:

```yaml
# ## 概要把握
# - 主要な主張: 45ヌクレオチドの小型RNAポリメラーゼリボザイム（QT45）が自身と相補鎖の両方を合成できる
# - 研究手法: ランダム配列プールからのde novo選択...
# - 主要な結果: QT45は共晶氷条件下でトリプレット基質を用い...
# - 理論的意義: ポリメラーゼ活性が小さなRNAモチーフでも実現可能...
```

これはHookで直接検証した項目ではない。しかしHookの存在がAI全体の手順遵守の意識を高めた可能性がある。あるいは単に今回の実行ではたまたま出力された可能性もある。確率的な効果であり、保証はされない。

## 4ステップの比較まとめ

| 観点 | Step 1 (CLAUDE.md) | Step 2 (+Rules) | Step 3 (+Skills) | Step 4 (+Hooks) |
|------|---|---|---|---|
| **追加した機能** | なし | フォーマット規則 | 手順定義 | 検証スクリプト |
| **制御の種類** | 確率的（最小） | 確率的（強化） | 確率的（手順化） | 決定論的 |
| **type体系** | 9種（自由発明） | 7種（固定） | 7種（固定） | 7種（検証済み） |
| **relation体系** | 自由な日本語 | 8種（固定） | 8種（固定） | 8種（検証済み） |
| **整合性** | 未検証 | 未検証 | 指示したが不履行 | スクリプトで保証 |
| **孤立ノード** | 未確認 | 1件 | 1件（修正指示無視） | 0件（強制修正） |
| **概要把握** | なし | なし | 指示したが不出力 | 出力された |

### 各ステップが解決した問題

- **Step 2 (Rules)**: フォーマットの不安定さ → type/relationの選択肢を固定
- **Step 3 (Skills)**: 手順の不在 → 抽出の手順を標準化（ただし効果は限定的）
- **Step 4 (Hooks)**: 整合性チェックの不履行 → コードで検証を強制

### 確率的制御と決定論的制御の境界

Step 2・Step 3で「守られた」ルールと、Step 4で「強制された」チェックの違い:

- **Rules/Skillsで守られたこと**: type 7種、relation 8種、snake_case命名、description簡潔化。これらは「守る確率が高い」が「保証はない」。今回たまたま100%遵守されたが、次回も100%である保証はない。
- **Hooksで強制されたこと**: ノード存在、エッジ存在、参照整合性、孤立ノードなし、type/relation許可リスト準拠。これらは「必ず検証される」。条件を満たさなければ完了が許可されない。

この境界線こそが本稿第2章で述べた「確率的制御と決定論的制御の使い分け」の実例であり、第5章で述べた「自由度の配置」の実践である。
