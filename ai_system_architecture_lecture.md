# AIシステム基盤構造 — LLMを「使いこなす」ための基礎知識

> **対象読者**: AIを日常的に使っているが、内部の仕組みは詳しくない方  
> **目的**: 「なぜAIは指示通りに動かないことがあるのか」を構造的に理解し、より効果的な使い方を身につける  
> **前提知識**: ChatGPT、Claude、Gemini等のAIチャットを使った経験があること

---

## はじめに — なぜ「仕組み」を知る必要があるのか

AIを使っていて、こんな経験はないだろうか。

- 「さっきはうまくいったのに、同じ指示で違う結果が出た」
- 「ルールを書いたのに、なぜか守ってくれないことがある」
- 「長い会話をしていたら、最初に言ったことを忘れた」
- 「昨日の会話の内容を覚えていない」

これらは「AIの性能が低い」のではなく、**AIシステムのアーキテクチャ（設計構造）に起因する、原理的な振る舞い**である。仕組みを知れば、これらは予測可能になり、対処できるようになる。

本稿では、LLM（大規模言語モデル）ベースのAIシステムの基盤構造を、ユーザーの視点から解説する。

### 本稿を読む前に — AIとの向き合い方

構造の話に入る前に、一つ重要なことを述べておく。

多くの人がAIを検索エンジンのように使っている。「〇〇とは何ですか」「〇〇のやり方を教えて」——答えを持っているAIから、正解を引き出そうとするアプローチだ。これはAIの能力の一部しか使っていない。

AIを使いこなす上で最も重要なのは**対話**である。答えを引き出すのではなく、**十分なインプットを持って、自分の思考をAIにぶつける**。AIはその思考に対して構造化、反論、補完、拡張を返す。それに対してまた自分の考えをぶつける。このサイクルの中で、最初には見えなかった構造や問題が浮かび上がってくる。

両者の違いを、同じテーマ——「AIに指示が通らない問題」——で比較してみよう。

```
受動的な使い方（授業モデル）:

  人間: 「AIシステムのアーキテクチャについて教えて」
  AI:   「AIシステムは、モデル層、オーケストレーション層、
         ガードレール層から構成され...（長い説明）」
  人間: （読む。なるほど。次の質問へ）
  人間: 「Hooksって何？」
  AI:   「Hooksとは、エージェントのライフサイクルイベントに
         紐づくスクリプト実行の仕組みで...（長い説明）」
  人間: （読む。なるほど。次の質問へ）
  人間: 「確率的制御と決定論的制御の違いは？」
  AI:   「確率的制御とは...決定論的制御とは...（長い説明）」

  → AIが話し、人間が聞く。教科書を読むのと変わらない。
     人間の頭の中にある疑問は解消されるが、新しい発見は生まれにくい。
```

```
能動的な使い方（対話モデル）:

  人間: 「Claude Codeでワークフローを組んでいるが、
         Rulesに『Phase毎にgit commitせよ』と書いても
         スキップされることがある。プロンプトの書き方が悪いのか？」
  AI:   「それはプロンプトの問題ではなく、アーキテクチャの問題だ。
         Rulesはコンテキストにテキストとして注入されるだけなので、
         モデルの確率的な処理でスキップされうる。
         100%の遵守が必要なら、Hooksで決定論的に制御すべき。」
  人間: 「なるほど、確率的制御の限界ということか。
         だとすると、全部Hooksにすればいいのでは？」
  AI:   「全部Hooksにすると、モデルの自由度が失われる。
         たとえばコード設計の判断までHooksで固めると、
         モデルの柔軟な問題解決能力が活かせなくなる。
         重要なのは『どこを固め、どこを自由にするか』の設計。」
  人間: 「それは自分の認知の問題でもある気がする。
         Claude.aiで設計を考えるときとClaude Codeで
         実装するときで、自分の頭の使い方が違う。
         混ぜると混乱する。」
  AI:   「興味深い指摘だ。ユーザー側のアテンションの分離と、
         AI側の確率的/決定論的制御の分離が、
         結果的に同じ構造になっている...」

  → 人間の具体的な経験から始まり、構造的な理解に到達し、
     さらに予想外の洞察（人間側の認知とAI側の構造の対応）が生まれる。
     これはどちらか一方だけでは辿り着けない。
```

二つの違いは明確だ。授業モデルでは、AIが持っている知識の範囲内で情報が提供される。対話モデルでは、人間が持っている文脈とAIの構造化能力が掛け合わさって、**どちらの頭の中にも最初はなかった発見**が生まれる。

実は、本稿自体がこの対話モデルで作られている。筆者がAIとの対話の中で「ステートレスとは何か」「Hooksはなぜ決定論的なのか」「Harnessとは何か」を順番に議論し、議論の過程で構造図や概念整理が生まれた。最初から「この内容を書こう」と決まっていたわけではなく、対話の中で構造が発見されていった。

なぜ後者が効果的なのかは、本稿で述べるアーキテクチャを理解すれば明確になる。先に結論だけ述べておくと、**AIの出力品質は、コンテキストウィンドウに何が入っているかで決まる**。「教えて」だけではコンテキストが貧弱で、汎用的な回答しか返せない。自分の思考、前提、文脈、判断根拠を十分にぶつければ、AIの応答精度は劇的に上がる。

## 第1章 ステートレス性 — AIは「覚えていない」

### 1.1 毎回が白紙からのスタート

LLMの最も重要な性質は**ステートレス（状態を持たない）**であることだ。

人間の脳は、昨日の会話を覚えている。神経回路の結合が物理的に変化して、記憶として定着するからだ。LLMはそうではない。モデルのパラメータ（数千億個の数値の集合）は、訓練が終わった時点で固定されている。ユーザーとの会話によってパラメータが書き換わることはない。

つまり、毎回のリクエストで起きていることはこうだ：

1. システムがコンテキストウィンドウを組み立てる（システムプロンプト＋過去のやり取り＋今回の入力）
2. その「テキストの塊」をモデルに丸ごと渡す
3. モデルはそれだけを見て応答を生成する
4. **生成が終わったら、モデル側には何も残らない**

### 1.2 「記憶」の正体

**同一会話内の「記憶」**

今の会話でAIがさっきの発言を踏まえて応答できるのは、「覚えている」からではない。会話の全履歴がコンテキストウィンドウに毎回入力として渡されているからだ。AIは毎ターン、会話の最初から全部読み直しているようなものである。

**会話をまたぐ「記憶」**

前回の会話の内容を踏まえた応答ができるのは、外部のシステムが記憶データベースから関連情報を検索し、今回のコンテキストウィンドウに注入しているからだ。モデルが「思い出している」のではなく、**「読まされている」**。

| | 人間の記憶 | LLMの「記憶」 |
|---|---|---|
| 保存場所 | 脳内（神経回路の変化） | 外部データベース |
| 想起の仕組み | 内部から取り出す | テキストとして入力に追加される |
| 永続性 | 脳が変化するので永続的 | 注入されなければ存在しない |

### 1.3 コンテキストウィンドウ — AIの「作業机」

コンテキストウィンドウとは、1回のやり取りでモデルが「見える」範囲のことだ。システムプロンプト、ユーザーの入力、過去のやり取り、ツール呼び出しの結果——これらすべてがこのウィンドウに収まっている必要がある。

**窓の外にあるものは、モデルにとって存在しない。**

会話が長くなってウィンドウの上限を超えると、古い部分が切り落とされる。これを auto-compact と呼ぶ。文字通り「見えなくなる」のであり、AIが「忘れた」わけではない——最初から知らない状態になる。

> **実用上のポイント**: 重要な指示や前提条件は、会話の途中で繰り返し伝えた方が確実である。「最初に言ったから覚えているはず」という前提は、アーキテクチャ上保証されない。

## 第2章 確率的制御と決定論的制御 — なぜ「指示通り」にならないのか

### 2.1 すべての出力は確率的である

LLMの生成プロセスは、トークン（単語の断片）を一つずつ順番に生成していく仕組みだ。各ステップで「次にどのトークンを出すか」は確率分布からサンプリングして決まる。

これが意味するのは、**プロンプトで「必ずXをしてからYをせよ」と書いても、モデルがその指示に従うかどうかは確率的だ**ということだ。99%従うかもしれないが、100%ではない。

ここが多くのユーザーが戸惑うポイントになっている。

### 2.2 二つのメンタルモデルの罠

多くの人がLLMに持ち込むメンタルモデルは、大きく二つに分かれる。

**「賢い同僚」モデル**: 指示すれば理解して従ってくれるはず。従わないのは指示が曖昧だから。→ 失敗するたびに指示を詳細にするが、それでも確率的に失敗する場面に遭遇して困惑する。

**「プログラム」モデル**: 正しいコマンドを与えれば確実に実行される。→ 同じ入力で異なる出力が出ることが受け入れられない。

**どちらも間違っている。** LLMは「指示を理解する確率的なシステム」であって、確実に従う同僚でも、決定論的に動くプログラムでもない。

### 2.3 ルール（確率的制御）の本質と限界

ユーザーがAIに与えるルールやプロンプト上の指示は、すべてコンテキストウィンドウにテキストとして入る。モデルの確率的生成プロセスに「影響を与える」ことはできるが、「強制する」ことはできない。

テキストベースの制御にも強弱のグラデーションがある：

| レベル | 影響力 | 例 |
|---|---|---|
| システムプロンプト | 最も強い確率的影響 | AI提供者が設定する基本指示 |
| Rules / 設定ファイル | 強い確率的影響 | セッション開始時に自動読み込み |
| コマンド内の指示 | 中程度 | 特定操作を呼び出した時のみ有効 |
| 会話中のユーザー指示 | 文脈依存 | 会話の流れの中で与える指示 |

**すべて確率的制御の枠内である。** 確率を上げることはできるが、保証はできない。

### 2.4 Hooks（決定論的制御）— 確率の外に出る

Hooksは、確率的な世界の外側にある**決定論的なコード実行**だ。

ルールの場合の処理の流れ:
```
プロンプトテキスト → モデルの注意機構 → 確率的生成 → 出力
（どの段階でもスキップされうる）
```

Hooksの場合の処理の流れ:
```
モデルの出力が生成される
  → オーケストレーション層がイベントを検知
  → モデルの外側でスクリプトが実行される
  → 結果に応じてモデルの次の行動が制御される
```

決定的な違いは、**Hooksの実行判断にモデルが関与しない**ことだ。if文は100%の確率で正しく分岐する。ここに確率的な揺らぎはない。

### 2.5 「完了」判断の構造

よくある誤解は「Hooksがあれば全部決定論的に制御できる」というものだ。実際にはそうではない。

モデルが「タスク完了です」と出力するかどうかは、確率的な判断だ。あと3ステップ残っているのに「完了」と宣言してしまうことは起こりうる。

Stop hookが保証しているのは、**「誤って完了と判断した場合に差し戻すこと」** だ：

```
モデル: 「完了です」（確率的 — 正しいかもしれないし、間違っているかもしれない）
    ↓
Stop hook: 条件チェック（決定論的）
    ↓
  条件未達 → 「まだ終わっていません」とモデルに返す
  条件達成 → 終了を許可
```

つまり、**「正しく完了させる」のではなく「間違った完了を阻止する」ゲート**である。

### 2.6 制御の二重構造 — 核心

```
確率的制御                          決定論的制御
(ルール・プロンプト)                (Hooks・コード実行)
─────────────────                   ─────────────────
モデルの内側で作用                  モデルの外側で作用
テキストとして注入                  コードとして実行
「影響を与える」                    「強制する」
遵守率 < 100%                       遵守率 = 100%
スケール容易                        設置コストあり
自由度を活かせる                    自由度を制限する

     │                                    │
     ▼                                    ▼
┌──────────────┐                   ┌──────────────┐
│ 適する場面    │                   │ 適する場面    │
│              │                   │              │
│ ・意図の解釈  │                   │ ・手順遵守    │
│ ・創造的生成  │                   │ ・状態検証    │
│ ・曖昧な指示  │                   │ ・品質ゲート  │
│ ・探索的作業  │                   │ ・必須記録    │
└──────────────┘                   └──────────────┘

     ▲                                    ▲
     │            設計判断                  │
     └──────── ここに何を置くか ────────────┘
               が使いこなしの鍵
```

> **実用上のポイント**: 「プロンプトの書き方」で解決しようとする前に、その問題が確率的制御の限界に起因するものではないか考えてみよう。100%の遵守が必要なら、ツールの選択や仕組みの設計を変える必要がある。

## 第3章 Harness と Context Engineering — AIシステムの設計論

### 3.1 用語の進化

AIを「うまく使う」ための概念は、ここ数年で急速に進化している。

**Prompt Engineering（プロンプトエンジニアリング）**: 指示文をどう書くかに注力する。最も狭い概念。

**Context Engineering（コンテキストエンジニアリング）**: コンテキストウィンドウの中身全体——システムプロンプト、メモリ、会話履歴、ツール結果のフォーマット——をどう設計するかに注力する。Prompt Engineeringを包含する、より広い概念。

**Harness Engineering（ハーネスエンジニアリング）**: モデルの外側のループ全体——LLM呼び出し、ツール呼び出し、バックグラウンドプロセス管理、コンテキスト取得、権限管理——をどう構成するかに注力する。Context Engineeringをさらに包含する、最も広い概念。

```
Harness Engineering（ループ全体の設計）
  ┃
  ┣━━ Context Engineering（コンテキストウィンドウの中身の設計）
  ┃     ┣ Prompt Engineering（指示文の設計）
  ┃     ┣ メモリ管理（短期/長期記憶の選別・注入）
  ┃     ┣ RAG / ツール結果のフォーマット
  ┃     ┗ → 確率的制御の質を最大化する
  ┃
  ┣━━ Architectural Constraints（決定論的制約）
  ┃     ┣ Hooks（Stop, PreToolUse, PostToolUse）
  ┃     ┣ リンター、構造テスト
  ┃     ┣ スクリプトによる検証
  ┃     ┗ → 決定論的制御を提供する
  ┃
  ┗━━ Garbage Collection（品質維持）
        ┣ 定期的な不整合検出
        ┣ ドキュメントの鮮度管理
        ┗ → システム全体のエントロピーとの戦い
```

### 3.2 Harness（ハーネス）とは何か

Harnessは元々「馬具」を意味する。馬の力を人間が制御し、有用な方向に導くための装備一式——手綱、くつわ、鞍、ベルトなどの総体だ。

AI文脈では、**LLMという強力だが確率的で予測困難な「馬」を、Hooks、ツール管理、コンテキスト制御、権限管理といった「馬具一式」で制御し、有用な仕事をさせる仕組み**のことだ。

重要なのは、馬具が馬の力を殺すものではないという含意だ。良い馬具は馬の能力を最大限に引き出しつつ、暴走を防ぐ。締めすぎれば馬は動けないし、緩すぎれば制御できない。

これは第2章で述べた「自由度の配置」そのものである。

### 3.3 Context Engineering が解決する問題

Context Engineeringが対象とするのは、コンテキストウィンドウの中身をどう組み立てるかだ。これは確率的制御の世界の話である。コンテキストの質を上げれば、モデルの判断精度は上がるが、保証はされない。

Context Engineeringの主な構成要素：

- **システムプロンプト**: モデルの振る舞いを定義する基本指示
- **メモリ管理**: 短期記憶（現在の会話履歴）と長期記憶（過去の会話から検索した情報）
- **RAG（検索拡張生成）**: 外部知識ベースからの情報取得と注入
- **ツール定義**: モデルが使えるツールの説明と使い方
- **出力フォーマット**: 期待する応答形式の指定

AIの応答品質が悪いとき、多くの場合はモデルの能力不足ではなく、コンテキストの問題である。必要な情報が欠けている、不要な情報でノイズが多い、情報の順序が不適切——これらを最適化するのがContext Engineeringだ。

### 3.4 対話はContext Engineeringの実践である

はじめにで述べた「思考をぶつける」というアプローチは、Context Engineeringの最も原始的な形だ。自分の頭の中にある文脈を、モデルのコンテキストウィンドウに転写する行為に他ならない。

効果的なインプットには以下の要素が含まれている。

**目的（Why）**: なぜこれをやりたいのか。「レシピデータを構造化したい」ではなく「レシピ間の関係性をネットワークとして可視化し、食材の組み合わせパターンを発見したい」の方が、AIは適切な構造を提案できる。

**文脈（Context）**: 現在の状況、制約条件、これまでに試したこと。「うまくいかない」ではなく「Phase 2まで完了したが、ノード数が500を超えた時点で可視化が破綻した」の方が、AIは的確な対処を返せる。

**自分の仮説（Hypothesis）**: 自分はどう考えているか。「どうすればいい？」ではなく「チャンクに分割して処理するアプローチが良いと思うが、重複ノードの処理が問題になりそうだ」の方が、AIはその仮説の妥当性を検証し、見落としを指摘できる。

**判断基準（Criteria）**: 何をもって成功とするか。「良い感じにして」ではなく「正確性が最優先、速度は二の次でよい。研究用なので再現可能性が重要」の方が、AIはトレードオフの判断を適切にできる。

これらの要素を対話の中でコンテキストウィンドウに蓄積することで、AIの出力精度は向上していく。ただしウィンドウには上限があるので、非常に長い対話では初期の文脈が失われることがある（第1章で述べたauto-compact）。これが起きたら、重要な結論をファイルに書き出す——外部永続化層の活用だ。

> **実用上のポイント**: 「AIが何をしてくれるか」ではなく「AIに何をぶつけるか」を考える。出力の質は入力の質で決まる。十分なインプットなしに良い出力を期待するのは、空のコンテキストウィンドウに高品質な応答を求めるようなものだ。

### 3.5 Harnessの境界 — 何が「馬具」で何が「馬」か

```
           Harness                        モデル
┌──────────────────────────┐    ┌──────────────────────────┐
│                          │    │                          │
│ オーケストレーション層   │    │ Transformer本体          │
│ Context Engineering      │    │ パラメータ（固定）       │
│ Hooks / ツール実行       │    │ 自己回帰生成             │
│ 外部ガードレール         │    │ 内部ガードレール         │
│ 外部永続化               │    │ (Constitutional AI等)    │
│                          │    │                          │
│ → ユーザーが設計可能     │    │ → ユーザーは変更不可     │
│ → Harness Engineering    │    │ → Model Engineering      │
│   の対象                 │    │   (AI提供者の領域)       │
└──────────────────────────┘    └──────────────────────────┘

馬具（装着・調整できる）         馬（そのものは変えられない）
```

ユーザーの立場からすると、**Harnessが自分の影響を及ぼせる範囲**であり、モデルはAI提供者（Anthropic、OpenAI等）の領域である。自分が設計・最適化できるのはHarnessの部分であり、モデルそのものを変えることはできない。

> **実用上のポイント**: 「AIの性能が悪い」と感じたとき、問題はモデルではなくHarnessにあることが多い。同じモデルでも、Harnessの設計次第で大きく性能が変わる。

## 第5章 実践への示唆 — 何をどう変えるか

### 5.1 よくある問題と、その構造的原因

| よくある不満 | 構造的原因 | 対処の方向性 |
|---|---|---|
| 同じ指示で違う結果が出る | 生成プロセスが確率的 | 温度パラメータの調整、出力形式の制約 |
| ルールを守ってくれない | テキスト指示は確率的制御 | Hooksの導入、または許容する設計 |
| 長い会話で最初の指示を忘れる | コンテキストウィンドウの上限 | 重要指示の繰り返し、外部ファイル化 |
| 昨日の話を覚えていない | ステートレス設計 | メモリ機能の活用、状態の外部永続化 |
| 途中で手順を飛ばす | 確率的に指示をスキップ | 決定論的チェックポイントの設置 |

### 5.2 自由度の配置という設計思考

LLMの出力が確率的であることは、弱点であると同時に強みでもある。曖昧な入力から意味のある出力を生成できるのは、この自由度のおかげだ。

重要なのは、自由度を一律に排除も放任もせず、**「どこで自由にして、どこで固めるか」を意識的に設計する**ことだ。

- **自由度を活かす場面**: アイデア出し、文章作成、探索的な調査、曖昧な意図の解釈
- **自由度を制限する場面**: 手順の遵守、データの正確性、フォーマットの統一、安全性の確保

この配分の判断こそが設計行為であり、プロンプトの文言を調整することよりもはるかに影響が大きい。

### 5.3 ツール選択の指針

現時点では、ユーザーが自分で決定論的制御を設計できるツールは限られている。

- **Claude.ai, ChatGPT等のチャットUI**: ほぼすべてが確率的制御。自由度の高い作業に向く。
- **Claude Code, Codex等のエージェントツール**: Hooksなどの決定論的制御を設計可能。手順遵守が重要な作業に向く。
- **API + 自作オーケストレーション**: 最も自由度の高いHarness設計が可能。ただし技術的コストが高い。

全部をチャットUIでやろうとするのではなく、タスクの性質に応じてツールを選ぶ——これも「自由度の配置」の一形態である。

### 6.2 CLAUDE.md — プロジェクトの「地図」

**アーキテクチャ上の位置**: Context Engineering層。コンテキストウィンドウに注入されるテキスト。  
**制御の種類**: 確率的制御

CLAUDE.mdは、セッション開始時に自動的にコンテキストウィンドウに読み込まれるMarkdownファイルだ。プロジェクトの概要、ディレクトリ構成、参照すべきファイル、作業上の前提条件などを記述する。

**実践例**: プロジェクトの案内板として使う

```markdown
# プロジェクト概要

レシピデータを構造化してネットワーク可視化するプロジェクト。

## ディレクトリ構成
- docs/     — 仕様書、進捗
- data/     — 入力データ（生レシピテキスト）
- output/   — 生成されたYAMLとHTML
- scripts/  — 検証・変換スクリプト

## 作業の前提
- 仕様変更は必ず docs/spec.md に記録する
- data/ 内のファイルは読み取り専用として扱う
```

**なぜ確率的制御なのか**: CLAUDE.mdはテキストとしてモデルに渡されるだけであり、「data/内のファイルは読み取り専用」と書いてあっても、モデルが書き込んでしまう可能性はゼロではない。重要度の高い制約は、後述のHooksで補強する必要がある。

### 6.3 Rules — 行動規則の定義

**アーキテクチャ上の位置**: Context Engineering層。CLAUDE.mdと同様にコンテキストに注入される。  
**制御の種類**: 確率的制御（ただしCLAUDE.mdより特化された指示）

Rulesは `.claude/rules/` ディレクトリに配置するMarkdownファイルで、モデルの行動規則を定義する。CLAUDE.mdがプロジェクト全体の「地図」なら、Rulesは「交通ルール」にあたる。

**実践例**: 作業規則と禁止事項

```markdown
# ファイル: .claude/rules/workflow.md

## 実行原則
- 作業開始前に必ず docs/progress.md を読み、現在の状態を確認する
- 1つのPhaseが完了するたびに progress.md を更新する

## 禁止事項
- spec.md に記載されていない機能を独自判断で追加しない
- 既存のテストを削除しない
- 作業対象外のディレクトリを変更しない
```

```markdown
# ファイル: .claude/rules/problem-handling.md

## 問題を発見した場合
- 作業を中断し、logs/issues.md に記録する
- 自己判断で修正せず、ユーザーに報告する
```

**確率的制御の限界が表れる場面**: Rulesに「spec.mdにない機能を追加するな」と書いても、モデルが「これは改善だ」と解釈して追加してしまうことがある。これは「ルールを無視した」のではなく、「ルールの解釈が人間と異なった」という確率的処理の結果だ。

### 6.4 Skills — 再利用可能なワークフロー定義

**アーキテクチャ上の位置**: Context Engineering層。呼び出し時にコンテキストに注入される。  
**制御の種類**: 確率的制御

Skillsは `.claude/skills/` ディレクトリに配置するワークフロー定義で、Claude Codeの拡張の中心的な仕組みだ。各Skillはフォルダとして構成され、`SKILL.md` を核に、テンプレートやスクリプトなどのサポートファイルをバンドルできる。

**Skillsの特徴**: `/スキル名` でユーザーが手動で呼び出せるだけでなく、タスクの内容に基づいてClaudeが自動的に発動させることもできる。これが旧来のSlash Commands（`.claude/commands/` に単体のMarkdownファイルを置く方式）との最大の違いだ。旧Commandsも後方互換で動作するが、同名の場合はSkillが優先される。現在はSkillsが推奨アプローチとなっている。

**実践例**: セットアップSkillと実行Skill

```
# ディレクトリ構成
.claude/skills/
  setup/
    SKILL.md          ← メインの指示（必須）
    templates/        ← テンプレートファイル
  execute/
    SKILL.md
    scripts/
      validate.sh     ← Skillから呼び出せるスクリプト
```

```markdown
# ファイル: .claude/skills/setup/SKILL.md

---
name: setup
description: プロジェクトの初期セットアップを行う。proposal.mdに基づいてspec.md、progress.md等を生成する。
---

## 手順
1. docs/proposal.md を読む
2. proposal.md の内容に基づいて以下のファイルを生成する：
   - docs/spec.md（仕様書）
   - docs/progress.md（進捗管理）
   - CLAUDE.md（プロジェクト指示）
3. 生成完了後「/setup 完了」と報告して停止する

## 禁止事項
- 実装（コードの記述・実行）に入らないこと
```

```markdown
# ファイル: .claude/skills/execute/SKILL.md

---
name: execute
description: spec.mdに従ってタスクを実行する。progress.mdで進捗を管理する。
---

## 手順
1. docs/progress.md を読み、現在のPhaseと未完了タスクを確認する
2. docs/spec.md を読み、タスクの詳細仕様を確認する
3. 未完了タスクを順番に実行する
4. 各タスク完了後に progress.md を更新する
5. Phase完了後に git commit する
```

**呼び出し方**: ターミナルで `/setup` や `/execute` と入力すると、SKILL.mdの内容がコンテキストに注入される。また、`description` フィールドの内容に基づいて、ユーザーが明示的に呼ばなくてもClaudeが適切なタイミングで自動的にSkillを選択・適用することがある。

**旧Commandsとの違い**:
- **Commands**（旧）: `.claude/commands/setup.md` のように単一ファイル。ユーザーが `/` で手動呼び出しするのみ。
- **Skills**（現在）: `.claude/skills/setup/SKILL.md` のようにディレクトリ構成。YAML frontmatterで名前と説明を定義。手動呼び出しに加えて自動発動にも対応。スクリプトやテンプレートをバンドル可能。

**Skillsの本質**: Skillsは「高品質なプロンプトの再利用」に加えて、「AIが自律的に適用できるワークフロー部品」である。スクリプトをバンドルすることで、確率的制御（SKILL.mdの指示テキスト）と決定論的制御（バンドルされたスクリプト）を一つのパッケージに統合できる。ただし、SKILL.md自体の指示に対する遵守は確率的であり、手順をスキップする可能性はある。

### 6.5 Hooks — 決定論的制御の実装

**アーキテクチャ上の位置**: Architectural Constraints層。オーケストレーション層のイベントループ内で動作。  
**制御の種類**: 決定論的制御

Hooksは `.claude/settings.json` で定義する。モデルの動作中の特定イベント（ツール使用前、ツール使用後、セッション終了前など）に、シェルスクリプトの実行を紐づける。

**実践例**: Stop Hook（タスク完了前の検証）

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "command": "bash .claude/hooks/check_completion.sh"
      }
    ]
  }
}
```

```bash
# ファイル: .claude/hooks/check_completion.sh

#!/bin/bash

errors=""

# progress.md が更新されているか確認
if ! git diff --name-only HEAD | grep -q "progress.md"; then
  errors="${errors}progress.md が更新されていません。\n"
fi

# git commit がされているか確認
if [ -n "$(git status --porcelain)" ]; then
  errors="${errors}コミットされていない変更があります。\n"
fi

if [ -n "$errors" ]; then
  echo "以下の問題があります："
  echo -e "$errors"
  echo "これらを解決してから完了してください。"
  exit 1  # 完了を拒否 → モデルに差し戻し
fi

exit 0  # 完了を許可
```

**なぜこれが決定論的なのか**: このスクリプトの実行にモデルは関与しない。`git diff` や `git status` は通常のプログラムとして動作し、条件分岐は100%正確だ。exit 1が返されればモデルの完了宣言は必ず却下される。モデルにこのチェックを「スキップ」する手段はない。

**実践例**: PreToolUse Hook（危険な操作の防止）

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "bash .claude/hooks/check_write_permission.sh \"$TOOL_INPUT\""
      }
    ]
  }
}
```

この例では、モデルがファイルに書き込もうとするたびに、書き込み先が許可されたディレクトリかどうかをスクリプトが検証する。RulesにもData/への書き込み禁止を書いているが、Hooksで二重に守ることで確実性が担保される。

### 6.6 MCP（Model Context Protocol）— 外部世界への接続

**アーキテクチャ上の位置**: Context Engineering層（ツール定義の提供）+ オーケストレーション層（ツール実行）  
**制御の種類**: ツール定義は確率的制御、ツール実行自体は決定論的

MCPは、AIモデルと外部ツール・データソースを接続するための標準プロトコルだ。ファイルシステムの操作、データベースへのアクセス、外部APIの呼び出しなどを、統一的なインターフェースで提供する。

```
モデル ←→ MCP ←→ 外部リソース

具体的な接続例：
  MCP Server: Filesystem  → ローカルファイルの読み書き
  MCP Server: GitHub      → リポジトリ操作
  MCP Server: Slack       → メッセージの送受信
  MCP Server: Database    → SQLクエリの実行
```

**MCPの二面性**: MCPには確率的な側面と決定論的な側面がある。モデルが「どのMCPツールをいつ呼ぶか」の判断は確率的だ。しかし、ツールが呼ばれた後の実際の実行（ファイルの読み書き、APIの呼び出し）は決定論的なコードとして動作する。

**実践例**: ファイルシステムMCPによるプロジェクトファイルへのアクセス

MCPサーバーを設定すると、モデルはチャットUIから直接プロジェクトのファイルを読み書きできるようになる。Claude.aiのプロジェクト機能でFilesystem MCPを使う場合、指定されたディレクトリ内のファイルを読み込んで作業の文脈として利用できる。

**MCPがHarnessの中で果たす役割**: MCPはモデルの「手足」を提供する。モデル単体はテキストを生成することしかできないが、MCPを通じて外部世界に作用できるようになる。これはHarnessのツール実行層の実装そのものだ。

### 6.7 機能の使い分け — 確率的/決定論的の配置

ここまでの機能を、制御の種類で整理する。

```
確率的制御（Context Engineering）
  ┃
  ┣━ CLAUDE.md    ── 常時読み込み。プロジェクトの前提。
  ┃                  最も広い範囲に影響するが、最も緩い制御。
  ┃
  ┣━ Rules        ── 常時読み込み。行動規則・禁止事項。
  ┃                  CLAUDE.mdより具体的な指示。
  ┃
  ┣━ Skills       ── 呼び出し時 or 自動発動。ワークフロー定義。
  ┃                  手順の一貫性を高める。スクリプトもバンドル可能。
  ┃                  指示テキスト部分はスキップされうる。
  ┃
  ┗━ MCP定義      ── ツールの説明。いつ何を使うかの判断。
                     モデルが使うかどうかは確率的。

決定論的制御（Architectural Constraints）
  ┃
  ┣━ Hooks        ── イベント駆動。条件チェック。
  ┃                  モデルがスキップできない強制力。
  ┃
  ┗━ MCP実行      ── ツール呼び出し後の実際の処理。
                     コードとして確実に実行される。
```

**実践上の設計パターン**: 重要な制約は「Rulesに書く＋Hooksでも守る」の二重構造にする。Rulesだけだと確率的に破られるリスクがある。Hooksだけだと、なぜその制約があるのかモデルが理解できず、不自然な動作をする可能性がある。両方を組み合わせることで、モデルは制約の意図を理解しつつ（Rules）、万が一忘れても強制される（Hooks）。

### 6.8 実証事例 — Hooksの有無が結果を決定的に変える

以下は、同じワークフローを3つの異なるツールで実行した比較実験の結果である。

| 指示内容 | Claude Code（Hooks有） | Codex CLI（Hooks無） | Gemini CLI（Hooks有） |
|---|---|---|---|
| Phase毎にgit commit | 全て実行された | **全てスキップされた** | 実行された |
| progress.md の更新 | 毎回更新された | まとめて事後記録 | 更新された |
| 手順の順序遵守 | 順番通り実行 | 一括実行・順序無視 | 順番通り実行 |

Codex CLIではHooksに相当する機構がないため、プロンプトに「Phase毎にcommitせよ」と何度書いても、全てスキップされた。これはプロンプトの書き方の問題ではなく、アーキテクチャの問題だ。モデルはプロンプトを「最終目標の記述」と解釈し、目標に直接関係しないステップ（commit、ログ記録）を自己判断で省略した。

この実験から得られた結論：

> **手順保証はプロンプトの工夫では達成できない。決定論的制御（Hooks）が必要。**  
> これは書き方の問題ではなく、LLMのアーキテクチャ的な限界である。

### 6.9 ワークフローの3層構造

実験を通じて、AIワークフローは以下の3層からなることが明らかになった。これは第4章の基盤構造図を、ユーザーが設計するワークフローの視点から再整理したものだ。

```
第1層: ドキュメント層 → ツール間で完全に共通
         spec.md, proposal.md, progress.md, issues.md
         → 外部永続化層に対応

第2層: 実行環境層   → 軽微な変換で移植可能
         Skills, Rules, エージェント定義
         → Context Engineering層に対応

第3層: 手順保証層   → 移植不可能（Hooks依存）
         Hooks（Stop, PreToolUse）
         → Architectural Constraints層に対応
```

第1層と第2層は確率的制御の世界にあり、ツール間で共有・移植できる。第3層は決定論的制御であり、その仕組みを持たないツールには移植できない。

### 6.10 実践 — Claude Codeでの開発プロセス

ここまでの機能が実際の開発でどう組み合わさるかを、一連の流れとして示す。

#### Phase 0: 構想を対話で練る（Claude.ai）

まず、チャットUIで構想を対話的に練り上げる。ここではHooksもSkillsも使わない。自由度の高い環境で、第3章で述べた「思考をぶつける」を実践する。

```
人間: 「レシピデータを概念ネットワークとして可視化したい。
       手元にレシピテキストが50件ある。
       各レシピから食材・調理法・味の方向性をノードとして抽出し、
       レシピ間の類似性をエッジとして表現したい。
       D3.jsの力学モデルで可視化する想定。」

AI:   「その構成だと、ノードの種類（食材/調理法/味）で
       3層構造になる。エッジの定義が重要で...
       5つの基準で関連度をスコアリングする方法がある...」

人間: 「5基準スコアリングは面白い。ただ50件で5基準だと
       組み合わせ爆発しないか？
       チャンク分割して処理すべきでは？」

AI:   「チャンク分割は有効。ただし分割境界での重複ノード処理が
       課題になる。chunk_{N}_プレフィックスで管理して
       後からmerge.pyで統合するアプローチが考えられる...」
```

この対話を通じて、プロジェクトの構想が具体化される。AIから「答えをもらう」のではなく、自分の構想をぶつけてAIの反応から構造を発見する。

#### Phase 1: proposal.mdの作成（Claude.ai）

対話で練った構想を、proposal.md（提案書）として構造化する。これは「対話の結晶」であり、第3章で述べたインプットの4要素（目的・文脈・仮説・判断基準）を文書化する作業だ。

```markdown
# proposal.md（抜粋）

## A. 目的
レシピデータの概念ネットワーク可視化

## B. 成功基準
- 50件のレシピからノード・エッジを抽出
- D3.js力学モデルで可視化
- ノード種別（食材/調理法/味）でフィルタリング可能

## D. 処理方針
- 10件ずつ5チャンクに分割して処理
- 5基準スコアリングでエッジ重み付け
- chunk_{N}_プレフィックスで重複管理

## F. Claude判断可の項目
- ノードIDの命名規則
- YAMLフォーマットの詳細
```

セクションFの「Claude判断可」が重要だ。ここに明示された項目はAIが自由に判断してよい。書かれていない項目は勝手に変更すべきではない。**自由度の配置を文書化する**ことで、確率的制御の範囲を明示的に設計している。

#### Phase 2: セットアップ（Claude Code）

ここから環境をClaude Codeに移す。proposal.mdに基づいてHarness一式を生成する。

```
人間: 「RUN.mdを読んで従って」

Claude Code:
  1. proposal.mdを読む
  2. ワークフローテンプレートからsetup手順を読む
  3. Harness一式を生成:
     - docs/spec.md, docs/progress.md
     - CLAUDE.md（6.2で解説）
     - .claude/skills/（6.4で解説）
     - .claude/rules/（6.3で解説）
     - .claude/settings.json + hooks/（6.5で解説）
  4. 「/setup 完了」と報告して停止
```

この時点で、本章で解説したすべての機能が配置される。馬に馬具を装着する作業だ。

#### Phase 3: 実行（Claude Code）

`/execute` コマンド（6.4）で実装を開始する。Hooks（6.5）が裏側で動き、各Phase完了時にprogress.md更新とgit commitを検証する。人間は見守るだけでよい。

#### Phase 4: レビューと対話

実装が完了したら、成果物を確認する。ここで再び「対話」に戻る。

```
人間: 「可視化結果を見たが、食材ノードが中心に集まりすぎている。
       ノード種別ごとに重力を変えるべきでは？」

AI:   「D3のforce simulationでカテゴリ別の引力を設定できる。
       ただしEdge bundlingを併用しないとエッジの交差が増える...」
```

修正が必要であればspec.mdを更新し、再度 `/execute` で修正を実行する。

#### 開発プロセスの全体像

```
Phase 0: 構想（Claude.ai — 対話）
  │  自由度：最大 ／ 制御：確率的のみ
  │  人間の役割：思考をぶつける、方向性を決める
  ▼
Phase 1: 提案書作成（Claude.ai — 対話 + 構造化）
  │  自由度：高 ／ 制御：確率的のみ
  │  人間の役割：意図を正確に文書化する → proposal.md
  ▼
Phase 2: セットアップ（Claude Code — コマンド実行）
  │  自由度：低 ／ 制御：確率的 + 決定論的（Hooks配置）
  │  人間の役割：実行を指示するだけ → Harness一式
  ▼
Phase 3: 実装（Claude Code — 自律実行）
  │  自由度：最低 ／ 制御：確率的 + 決定論的
  │  人間の役割：見守る（必要時のみ介入）
  ▼
Phase 4: 評価（対話）
  │  自由度：最大 ／ 制御：確率的のみ
  │  人間の役割：最終判断、フィードバック
  ├── 満足 → 完了
  └── 修正要 → Phase 1 or 3 へ戻る
```

このプロセス全体を貫いているのは**自由度のグラデーション**だ。最初は自由度が最大で、セットアップで徐々に固まり、実装では最小になり、評価でまた最大に戻る。そしてどのフェーズでも、人間の仕事は「思考をぶつける」ことだ。Phase 0では構想を、Phase 1では意図を、Phase 4では評価をぶつける。AIの仕事は、ぶつけられた思考を構造化し、実行し、結果を返すことだ。

## おわりに — 思考をぶつけること、構造を知ること

本稿で述べた内容を要約するなら、二つの軸がある。

**構造の理解**: LLMの出力はすべて確率的であり、プロンプトによる制御には原理的な限界がある。その限界を補うために、モデルの外側に決定論的な制御（Harness）を設計する。

**対話の実践**: AIを検索エンジンとして使うのではなく、十分なインプットを持って思考をぶつける。出力の質は入力の質で決まる。コンテキストウィンドウに何を載せるかが、すべてを左右する。

多くの人がAIに対して感じる「不満」や「不信感」の多くは、このアーキテクチャの性質を知らないことに起因している。「AIが悪い」のでも「自分のプロンプトが悪い」のでもなく、確率的システムに決定論的な結果を期待していた——というミスマッチが原因であることが多い。

この構造を理解した上で、確率的制御と決定論的制御の境界線をどこに引くか、そしてAIに何をぶつけるか——それがAIを「使いこなす」ための出発点になる。
